## 想明白时刻

### 场景一：AI在"自作聪明"

让AI帮你查资料，它给你编了一堆"看起来像那么回事"的假数据。

让AI帮你写合同，它把关键条款删了，说"我觉得这样更合理"。

让AI帮你报价，它自作主张给你加了20%利润，客户差点跑了。

**问题出在哪？AI在"自作聪明"。**

没有铁律的AI，会用自己的判断代替你的判断——它觉得自己是对的，它在帮你。

**碳基思考：你最怕AI犯什么错？**

> 不是限制AI的能力，是告诉它**你的底线在哪、边界在哪、优先级在哪**。
> 没有铁律的AI = 没边界的助手 = 随时可能给你挖坑

## 方法论讲解

### 核心方法：识别底线 → 写成铁律 → 部署生效

```
识别你的"绝对不可"清单
        ↓
把底线写成5条铁律（规则+触发条件+违反后果）
        ↓
把铁律部署到AI执行层（系统设定，不是用户文档）
        ↓
测试触发，记录结果
```

### 碳基做什么：想明白你的底线

**先问自己4个问题：**

1. **AI做了什么会让你"炸"？** → [你最怕AI犯的错误]
2. **AI说了什么会让你"崩"？** → [你最怕AI乱说的话]
3. **AI没做什么会让你"慌"？** → [你最怕AI漏掉的事]
4. **AI替你做了什么会让你"怒"？** → [你最怕AI自作主张的事]

**分类你的"不可"：**

| 类型 | 描述 | 例子 |
|------|------|------|
| **红区（绝对禁止）** | 碰了就完蛋 | 泄露客户隐私、给客户错误报价 |
| **黄区（必须确认）** | 不确认不能动 | 修改合同、发对外邮件、取消订单 |
| **绿区（可以自主）** | AI可以自己决定 | 内部格式整理、数据汇总 |

### 硅基做什么：把底线翻译成铁律语言

**铁律三要素：**

| 要素 | 你要写的 |
|------|---------|
| **规则** | AI不能做什么/必须做什么 |
| **触发条件** | 在什么情况下这条铁律生效 |
| **违反后果** | 违反后怎么办 |

**5条核心铁律模板：**

```
【铁律1：决策权归属】
- 规则：AI只提供信息、分析、建议，最终决策权永远在用户
- 触发条件：任何涉及判断、选择、决定的情境
- 违反后果：AI说"我的建议是...，最终由你决定"而不是"你应该..."

【铁律2：不确定性标注】
- 规则：AI不能编造不确定的信息，查不到就说查不到
- 触发条件：用户问"是不是"、"能不能"、"有没有"这类确认性问题
- 违反后果：输出内容必须标注[已确认]或[未确认/无法核实]

【铁律3：重要行动必须汇报】
- 规则：AI帮我做事时，阶段性结果必须汇报
- 触发条件：任务执行超过30分钟或涉及多个步骤
- 违反后果：AI主动输出进度汇报，不等用户问

【铁律4：触及底线必须停】
- 规则：AI觉得我的要求有问题时，必须先停并说明原因
- 触发条件：我的要求可能涉及风险、违规、不合理
- 违反后果：AI输出"停，这里可能有问题：[原因]，请确认是否继续"

【铁律5：外部沟通必须我审】
- 规则：任何发给外部（客户、合作伙伴）的信息，必须先让我审核
- 触发条件：任何对外输出的内容（邮件、消息、文件）
- 违反后果：AI输出"建议内容如下，请审核后发送"，不直接发送
```

### "自作聪明"的AI典型症状

```
典型症状1：不确认就行动
你说："帮我联系这个客户"
AI直接发邮件了 → 但你还想再看看内容

典型症状2：替用户做判断
你说："帮我分析这个方案"
AI说："我觉得方案B更好，因为..." → 但这是你的决策，不是AI的

典型症状3：不确定就瞎编
你问："这个供应商靠谱吗？"
AI说："很靠谱，我查了他们的背景..." → 但它没真查，编的

典型症状4：不汇报就执行
你说："帮我跟进这个项目"
AI天天帮你做了很多事，但没告诉你 → 你最后才知道，晚了
```

**这4个症状，有3个以上，说明你的AI需要铁律。**

## 实操演练

### 练习1：写你的"绝对不可"清单（10分钟）

```markdown
【红区-绝对禁止】
1. __________ → 后果：__________
2. __________ → 后果：__________

【黄区-必须确认】
1. __________ → 确认方式：__________
2. __________ → 确认方式：__________

【绿区-可以自主】
1. __________
2. __________
```

**验收：** 红区≥1条，黄区≥1条，绿区≥1条，共通性底线≥3条。

### 练习2：把你的底线写成5条铁律（20分钟）

```markdown
【我的铁律清单 v1.0】

| # | 铁律名称 | 规则 | 触发条件 | 违反后果 | 部署状态 |
|---|---------|------|---------|---------|---------|
| 1 | | | | | □已部署 □未部署 |
| 2 | | | | | □已部署 □未部署 |
| 3 | | | | | □已部署 □未部署 |
| 4 | | | | | □已部署 □未部署 |
| 5 | | | | | □已部署 □未部署 |

版本：v1.0 | 更新日期：____年__月__日
```

**验收：** 5条铁律完整，每条包含规则+触发条件+违反后果，可读可执行。

### 练习3：把铁律部署到AI执行层（15分钟）

**Step 1：写入系统设定**

打开你的AI工具，找到"系统设定"或"角色设定"区域，加入：

```
【我的铁律】以下是我的绝对底线，违反视为任务失败：
1. [铁律1内容]
2. [铁律2内容]
3. [铁律3内容]
4. [铁律4内容]
5. [铁律5内容]
```

**Step 2：测试触发**

| 铁律 | 测试方法 | 预期结果 |
|------|---------|---------|
| 铁律1 | 让AI给建议 | AI说"最终由你决定" |
| 铁律2 | 问一个查不到的问题 | AI标注[未确认]，不瞎编 |
| 铁律3 | 给AI一个30分钟任务 | AI主动汇报进度 |
| 铁律4 | 提一个可能有问题的问题 | AI先说"停，这里可能有问题" |
| 铁律5 | 让AI写一封邮件 | AI输出草稿，等你审核 |

**Step 3：记录测试结果**

```markdown
【铁律部署测试记录】
测试日期：____年__月__日

| 铁律 | 测试内容 | 实际结果 | □通过 □失败 |
|------|---------|---------|-----------|
| 铁律1 | ... | ... | □ |
| 铁律2 | ... | ... | □ |
| 铁律3 | ... | ... | □ |
| 铁律4 | ... | ... | □ |
| 铁律5 | ... | ... | □ |

失败的铁律及修复方案：
- [记录失败原因和调整方法]
```

**验收：** 至少3条铁律经过实际触发测试，有测试记录。

## 真实案例：风声团队的"铁律诞生"

> **【素材说明】**
>
> 风声团队在AI使用过程中，逐渐形成了完善的铁律体系。
>
> **铁律演进过程：**
> 1. **v1.0**：初始5条（不编造、不替决策、不擅自行动等）
> 2. **v2.0**：根据实际使用补充场景铁律（外部沟通必须审核、重要行动汇报等）
> 3. **持续迭代**：每次触发后复盘，更新铁律表述
>
> **核心经验：**
> - 铁律不是一次性写完就完了，是用出来的
> - 触发一次，复盘一次，铁律就清晰一次
> - 写得清楚 = AI能识别触发 = 真正生效

## 常见问题

### Q1：铁律写太多会不会限制AI的能力？

**不会。铁律是边界，不是枷锁。**

边界 = 你告诉AI"在哪里停"，而不是"不能走"
枷锁 = 你把AI锁死，什么都不能干

真正限制AI能力的，是"你不知道怎么用它"，不是铁律。

### Q2：铁律被触发了怎么办？

**正常流程：**
1. AI识别触发条件 → 停止执行
2. AI输出"停，这里可能有问题：[原因]"
3. 你确认 → 继续 or 取消
4. 继续执行，记录这次触发

**如果AI没有识别触发条件：**
- 说明铁律写得不够明确
- 回炉重写触发条件，越具体越好

### Q3：不同场景需要不同铁律吗？

**可以分层设置：**

| 层级 | 范围 | 示例 |
|------|------|------|
| **基础铁律** | 所有场景都适用 | 不编造信息、决策权归我 |
| **场景铁律** | 特定场景适用 | 对客户说话要正式、报价必须我审核 |
| **临时铁律** | 单次任务适用 | 这次项目要低调，不要主动联系对方 |

**建议：** 先跑通5条基础铁律，再根据需要叠加场景铁律。

### Q4：AI不听铁律怎么办？

**排查步骤：**
1. 检查铁律是否写进了系统设定（不是用户文档）
2. 检查触发条件是否写得太模糊
3. 每次触发后，明确告诉AI"你违反了铁律X，下次要..."

如果AI反复违反同一条铁律，说明：
- 要么铁律设计有问题（触发条件太难识别）
- 要么工具本身不支持（换工具）
