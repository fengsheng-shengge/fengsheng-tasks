# 数字员工交付交接文档模板

> **用途**：所有数字员工完成开发/迭代任务后，必须按此模板输出交接文档
> **原则**：给审核者看得懂、给部署者能照做、给后续者能接手
> **推广日期**：2026-07-21（源自小扣子V2.0交接文档，生哥确认可推广）

---

## 模板结构（6大板块，缺一不可）

```
一、本次交付内容
二、构建/测试状态
三、给审核者的清单（小酒窝）
四、给部署者的清单（小鱼儿）
五、后续规划（不在本次范围）
六、Git 提交记录
```

---

### 一、本次交付内容

**格式要求**：
- 按优先级分 P0/P1/P2，每个条目列出：
  - ✅ 完成状态
  - 涉及文件路径
  - 功能说明（1-3句）
  - 依赖项
  - 埋点/接口等外部关联
- 未完成项标注 ❌ + 原因 + 预计时间

**示例**：
```
#### P0-01：首页"客户说了什么"触发入口 ✅
- 文件：src/pages/home/index.vue
- 功能：输入客户原话 → 关键词匹配 → 展示结果
- 依赖：search_aliases.json（241条别名映射）
- 埋点：3个事件已接入
```

---

### 二、构建/测试状态

**格式要求**：
- 每个目标平台一行（mp-weixin / H5 / API 等）
- 构建结果：✅成功 / ❌失败+错误信息
- 关键页面/功能逐项验证，附截图或日志摘要

**示例**：
```
| 平台 | 状态 | 说明 |
|------|------|------|
| mp-weixin | ✅ 构建成功 | dist/build/mp-weixin/ 可导入 |
| H5 | ✅ 运行正常 | 端口5175 |

验证：12页全部渲染正常 ✅
```

---

### 三、给审核者的清单

**核心原则**：审核者（小酒窝）看这一节就能完成品控，不需要读全部代码

**格式要求**：
1. **审核重点**：列出本次修改的核心文件 + 审核关注点
2. **审核签字清单**：可勾选的 checklist，审核者逐项打勾
3. **已知限制/风险**：提前说明，减少审核者踩坑

**示例**：
```
### 审核重点
1. tracker.js：事件命名规范 + 字段完整性
2. 首页触发入口：匹配逻辑 + 跳转逻辑
3. 时效标注：srcType判断逻辑 + 已废止法条处理

### 审核签字
- [ ] 事件设计合理
- [ ] 匹配逻辑正确
- [ ] 时效标注无误
- [ ] 无语法错误/无运行时警告
```

---

### 四、给部署者的清单

**核心原则**：部署者（小鱼儿）照做就能上线，不需要问任何人

**格式要求**：
1. **部署步骤**：可复制粘贴的命令序列
2. **上传/发布步骤**：平台操作指引（AppID/版本号/备注）
3. **部署后检查项**：上线后必须验证的点，可勾选
4. **注意事项**：已知限制、环境差异、临时方案等

**示例**：
```
### 部署步骤
```bash
git pull origin main
npm install
npm run build:mp-weixin
```

### 上传
1. 微信开发者工具导入 dist/build/mp-weixin
2. AppID：wxd0f79d52a84ea4a8
3. 版本：2.0.0

### 部署后检查
- [ ] 首页触发入口正常
- [ ] 埋点数据写入正常
- [ ] 登录页功能列表正确
```

---

### 五、后续规划（不在本次范围）

**格式要求**：
- 列出已知但本次不做的项
- 每项标注：编号 + 内容 + 前置依赖
- 防止"忘了做"和"不知道还要做"

---

### 六、Git 提交记录

**格式要求**：
- 每个相关commit一行：hash + 说明
- 方便追溯和回滚

---

## 执行规则

1. **谁写**：任务执行者（开发/迭代的数字员工）
2. **何时写**：任务完成、测试通过后，推送前写
3. **存放位置**：项目 `docs/` 目录，命名格式 `V{版本}_交接文档_{审核者}_{部署者}.md`
4. **推送到GitHub**：交接文档随代码一起推送
5. **通知**：写完后 @审核者 + @部署者

## 角色映射

| 数字员工 | 常见角色 |
|---------|---------|
| 小扣子 | 开发执行者（写交接文档） |
| 小豆子 | 产品设计者（出需求文档） |
| 小酒窝儿 | 审核者（按清单品控） |
| 小鱼儿 | 部署者（按清单上线） |
| 小眼镜 | 知识输出者（内容审核） |
