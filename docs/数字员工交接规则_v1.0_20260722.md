# 风声数字员工上下游交接规则

> v1.0 | 2026-07-22 | 生哥确立
> 动因：7.22部署互相覆盖（小鱼儿覆盖小扣子_worker.js，小扣子覆盖小鱼儿verify Function）+ SA补充9天无人合并

---

## 一、总原则

**每次交接=交付物 + 验证结果 + 影响范围声明。三缺一不算交接完成。**

交接不是"我做了你接"，是"我做完告诉你我改了什么、影响什么、你接之前验什么"。

---

## 二、部署交接（最高频·最易出事）

### 2.1 部署权限（铁律·不可违反）

| 角色 | 权限 | 说明 |
|------|------|------|
| **小鱼儿** | ✅ 可部署 | 唯一部署执行人 |
| **小扣子** | ❌ 无部署权 | 只push代码+做部署准备，通知小鱼儿执行 |
| **小豆子** | ❌ 无部署权 | 产出设计稿/代码，通知小鱼儿或小扣子 |
| **小酒窝儿** | ❌ 无部署权 | 品控审核通过后才可部署 |
| **小眼镜** | ❌ 无部署权 | 知识产出push到仓库，通知小鱼儿部署 |

> **铁律：任何wrangler/cloudflare部署操作，只有小鱼儿执行。其他人push代码后必须通知小鱼儿，由小鱼儿统一部署并验证。**

### 2.2 部署前必检（部署者 checklist）

每次部署前，小鱼儿必须完成：

- [ ] **拉最新代码**：`git pull` 确保包含所有人的最新提交
- [ ] **检查functions目录**：确认Pages Function文件完整（不被覆盖）
- [ ] **检查_worker.js**：确认Worker代码与仓库一致（不被旧版覆盖）
- [ ] **检查D1 binding**：确认部署配置包含D1数据库绑定
- [ ] **记录部署前基线**：curl验证关键API状态，记录当前返回

### 2.3 部署后必验（部署者 checklist）

每次部署后，小鱼儿必须完成：

- [ ] **verify文件**：微信校验文件精确32字节
- [ ] **API全绿**：6个端点逐一验证（health/summary/daily/event/feedback/mentor-health）
- [ ] **页面可访问**：首页+关键页200
- [ ] **D1连通**：health返回connected
- [ ] **对比基线**：与部署前基线对比，确认无退化

### 2.4 部署冲突防护

| 场景 | 规则 |
|------|------|
| A推了代码，B也推了代码 | 小鱼儿部署前git pull合并，两人代码都包含 |
| A改了_worker.js，B改了functions/ | 部署时两者都会编译进Worker，不冲突 |
| 部署后发现问题 | 小鱼儿10分钟内定位+修复+重部署，不回滚（保持最新） |
| CI/CD vs 手动部署 | 8月前CI/CD额度耗尽，一律手动部署；8月后CI/CD优先，手动兜底 |

## 三、任务交接（上下游流转）

### 3.1 交接三件套

每个任务从A交到B，必须包含：

| 项 | 定义 | 示例 |
|----|------|------|
| **交付物** | A产出的具体文件/代码/文档 | entries.json（42条SA补充） |
| **验收条件** | B接手前怎么判断A的产出合格 | JSON验证通过+ID匹配entries |
| **影响范围** | 改了什么、会影响谁 | 改了entries.json，影响知识页面和小程序 |

**缺任何一项，B有权拒接。**

### 3.2 典型交接链路

```
产品设计(小豆子) → 代码实现(小豆子/小扣子) → 品控审核(小酒窝儿) → 部署(小鱼儿) → 验证(小鱼儿)
     ↓                    ↓                      ↓                    ↓
   交付物=PRD/原型     交付物=代码+自测        交付物=审核报告      交付物=线上验证结果
   验收=生哥评审       验收=编译通过+单测      验收=P0全过          验收=7项全绿
```

知识产出链路：
```
词条编写(小眼镜) → 条目补充(小眼镜) → 合并部署(小鱼儿) → 品控抽查(小酒窝儿)
     ↓                    ↓                  ↓
   交付物=entries       交付物=sA补充文件    交付物=线上可查
   验收=JSON格式通过    验收=ID匹配+内容合规  验收=SA率达标
```

### 3.3 交接格式（写入每日工单）

```markdown
## 交接 → [接手人]
- **交付物**：[文件路径或具体内容]
- **验收条件**：[怎么判断合格]
- **影响范围**：[改了什么影响谁]
- **Deadline**：[最晚什么时候完成]
- **阻塞项**：[有什么卡点需要先解决]
```

## 四、逾期升级规则

逾期不是"等"，是有明确升级路径的。

| 逾期时长 | 动作 | 执行者 |
|----------|------|--------|
| **到期日+0** | 任务标记⚠️，提醒负责人 | 工单系统自动 |
| **到期日+1天** | 升级小鱼儿，评估是否替补介入 | 小鱼儿 |
| **到期日+2天** | 🔴升级生哥，决策：替补/降级/取消 | 小鱼儿 |
| **到期日+3天** | 记入执行力档案，影响五把尺子评分 | 小鱼儿 |

**替补规则**：
- 同平台优先顶（小扣子逾期→小鱼儿直接修代码部署）
- 替补只保交付，不做方向决策
- 替补完成后复盘原负责人，找根因防再犯

## 五、越界防护（各人职责边界）

### 5.1 能做与不能做

| 角色 | 能做 | 不能做 |
|------|------|--------|
| 小鱼儿 | 部署、全局调度、修代码应急、跨Agent协调 | 代替小豆子做产品设计、代替小眼镜写词条 |
| 小扣子 | 测试、修Bug、push代码、部署准备 | 直接部署上线、做产品设计决策 |
| 小豆子 | 产品方案、UI/UX设计、前端代码实现 | 部署上线、品控审核 |
| 小酒窝儿 | 品控审核、VI检查、合规检查 | 修改代码、做产品决策 |
| 小眼镜 | 词条编写、知识运营、数据分析 | 代码修改、部署操作 |

### 5.2 补位例外

当逾期升级触发替补时，替补者临时获得越界权限，但：
1. 只做应急交付，不改架构/方向
2. 交付后24小时内复盘，确认原负责人恢复
3. 补位动作记录在案，不常态化
