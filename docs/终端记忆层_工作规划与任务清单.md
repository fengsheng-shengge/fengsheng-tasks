## 总体目标

按四层联动架构（长期记忆→工作记忆→经验记忆→情境记忆）搭建到终端最小可用，逐步扩展到AI眼镜可接入。

## P0 · 7月22-31日 · 支撑小程序V2上线

### T01 · trade域4张场景卡片+详情页
- **交付物**：4张场景卡片JSON（cardId/scenarioTitle/painPoint/professionalTake/legalBasis/relatedCases/relatedTools/tags）+ 4张详情页（legalBasis数组+cases数组+tools数组+brokerNote）
- **4个场景**：SCENE-05 定金纠纷 / SCENE-06 学区房承诺 / SCENE-07 阴阳合同 / SCENE-08 中介费争议
- **标准**：每张卡片关联3-5张工具卡 + 2-3条案例 + 法条依据；每条案例含caseId/title/judgment/direction/professionalInsight
- **依赖**：需要trade域工具卡已有（T02完成后回填关联）
- **产出路径**：`智能工具夹/scene_cards_v2.json` → GitHub `data/scene_cards.json`
- **截止**：7月25日

### T02 · clientCard字段扩展（3→7字段）
- **交付物**：broker_toolkit_rental_trade_v3.0.json，68条工具卡的clientCard从3字段扩展为7字段
- **新增字段**：
  - `keyPoints`：3-5个关键点（数组）
  - `legalBasis`：法律依据（hard词条必填，soft可为空）
  - `relatedCaseIds`：关联案例ID数组
  - `lawLayer`：规律锚定（市场规律/法律规律/人性规律至少一层，已在顶层存在但clientCard内也需标注）
- **验证**：每张卡走完"词条→工具卡→案例→法条"链路
- **产出路径**：`智能工具夹/broker_toolkit_rental_trade_v3.0.json` → GitHub `data/broker_toolkit.json`
- **截止**：7月27日

### T03 · 高频卡点场景清单
- **交付物**：一份Markdown清单，50-80个经纪人高频卡点场景，每个场景标注：所属域、是否有词条覆盖、词条ID、是否已有工具卡、是否已有案例
- **来源**：从241条词条的consumerQ提取 + 房产经纪行业经验补充
- **用途**：指导后续词条生产的优先级
- **产出路径**：`智能工具夹/高频卡点场景清单.md` → GitHub `docs/高频卡点场景清单.md`
- **截止**：7月29日

### T04 · 68条搜索别名→场景卡片→工具卡链路验证
- **交付物**：验证报告，确认每条搜索别名都能走完"搜索词→场景卡片→工具卡→词条→案例"链路
- **断链修复**：发现链路断裂的，补齐中间节点
- **产出路径**：`智能工具夹/搜索链路验证报告.md`
- **截止**：7月31日

## P1 · 8月1-15日 · 支撑AI员工+游戏化

### T05 · policy域40条工具卡映射
- **交付物**：policy域40条词条的工具卡JSON，7字段完整版
- **优先级**：severity=hard的先做（法条密集，AI员工最需要）
- **标准**：同T02的7字段标准，每张卡必须有toolType和lawLayer
- **产出路径**：`智能工具夹/broker_toolkit_policy_v1.0.json` → 合并入 `data/broker_toolkit.json`
- **截止**：8月5日

### T06 · 触发词索引建设（主动触发P0）
- **交付物**：trigger_words.json，400-500个触发词
- **来源**：241条词条的consumerQ + 68条search_aliases + 高频卡点场景清单
- **字段**：word / sceneId / toolIds / priority / context
- **标准**：50个高频卡点场景中，40个以上能被触发词命中（≥80%覆盖率）
- **后续对接**：小程序搜索增强的智能推荐 + AI眼镜语音触发
- **产出路径**：`智能工具夹/trigger_words.json` → GitHub `data/trigger_words.json`
- **截止**：8月3日

### T07 · 对话流索引建设（主动触发P0）
- **交付物**：dialog_flow_map.json，8个场景域各3-5条典型对话路径
- **内容**：每条路径5-10个节点，每个节点定义触发条件+推送记忆+后续节点
- **优先**：rental 4场景+trade 4场景（与场景卡片对齐）
- **标准**：每条路径从"客户开口"→"问题解决"闭环，无断链
- **产出路径**：`智能工具夹/dialog_flow_map.json` → GitHub `data/dialog_flow_map.json`
- **截止**：8月8日

### T08 · 案例采集流水线搭建
- **交付物**：
  1. 采集源清单（裁判文书网/法院公告/住建部通报/中国消费者报等5-8个A级来源）
  2. 采集模板（caseId/title/court/date/source/sourceUrl/direction/judgment/professionalInsight/relatedToolIds/relatedEntryIds）
  3. CodeAct脚本：每周自动检索关键词，产出候选案例清单
- **采集节奏**：每周10条候选，人工审核后入库
- **产出路径**：`智能工具夹/案例采集流水线.md` + `codeact/scripts/case_collector.py`
- **截止**：8月8日

### T07 · 客户模拟器V2（8→20场景）
- **交付物**：game_client_simulator_v2.json，20个场景
- **新增场景**：rental再补4个（合租纠纷/房屋损坏/提前退租/租金贷）+ trade 8个（定金纠纷/学区房/阴阳合同/中介费争议/户口迁出/购房资格/贷款审批/交房验收）
- **标准**：每个场景4级评分（methodology/halfMethod/script/wrong），与场景卡片1:1对齐
- **依赖**：T01的4张trade场景卡片完成后对齐
- **产出路径**：`智能工具夹/game_client_simulator_v2.json` → GitHub `data/game_client_simulator.json`
- **截止**：8月12日

### T08 · hard词条案例补强（第一批）
- **交付物**：至少20条新案例入库，优先补68条rental+trade工具卡中hard词条的案例缺口
- **标准**：A级来源，正负均衡（正向≥30%），每条含professionalInsight
- **来源**：T06流水线产出 + 定向搜索补强
- **产出路径**：`智能工具夹/case_library_v3.json` → GitHub `data/case_library.json`
- **截止**：8月15日

## 风险与卡点

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 案例采集效率低（裁判文书网搜索不稳定） | 经验记忆增长慢 | 多源采集（法院公告+住建通报+消费者报+律师公号） |
| 小酒窝儿审核排队 | 交付链路阻塞 | 提前48小时提交审核，预留返工时间 |
| 小扣子开发排期与内容交付时间差 | 内容到位但产品没接上 | 按小豆子开发排期倒推内容交付节奏 |
| 非rental/trade域consumerQ质量参差 | 工具卡映射质量不高 | 先做高频卡点清单（T03），优先映射高频场景 |
