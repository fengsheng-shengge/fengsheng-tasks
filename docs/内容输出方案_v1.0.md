## 五、技术对接说明

### 5.1 数据格式
- 所有内容文件统一JSON格式
- 存放路径：GitHub `fengsheng-shengge/fengsheng-tasks` 仓库 `data/` 目录
- 小程序通过API拉取，支持热更新

### 5.2 字段映射（entries.json → 小程序）

| 小程序页面 | 需要的entries字段 | 当前覆盖率 |
|-----------|-----------------|-----------|
| 词条标题 | title | 100% |
| 一句话结论 | oneLineAnswer | 28% (68/241) |
| 客户问题 | consumerQ | 28% |
| 客户收益 | consumerBenefit | 28% |
| 法律依据 | legalSources | 14% (34/241) |
| 关联案例 | caseLibraryIds | 5% (13/241) |

**缺口主要集中在非rental/trade域**（policy/decor/homekeep等173条暂无consumerQ等字段）。
P0先交付rental+trade 68条，P1再扩展其他域。

### 5.3 游戏化对接
- 客户模拟器和法官来了共用同一套评分引擎
- 每日一题从游戏JSON中按日期轮换取出
- 用户答题记录存本地，后续可同步服务端做排行榜

## 六、与小鱼儿同步要点

1. **内容铁律**：所有内容客观中立，不加主观评判（生哥明确要求）
2. **来源分级**：A级可直接引用，B级标注"需交叉验证"，不收录C级
3. **正向优先**：案例库正负均衡，不堆惨案恐吓客户
4. **方法论>话术**：游戏评分机制体现——专业佐证100分，纯话术30分
5. **产品定位**："经纪人智能工具夹"——方法论+佐证+案例，不是话术库
6. **数据源铁律**：entries.json是词条唯一数据源，禁止碎片拼凑
