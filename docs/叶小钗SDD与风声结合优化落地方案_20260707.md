## 五项可立即推进的结合优化

### 优化1：entries.json Schema扩展——补入"约束"和"验收标准"

**叶小钗依据**：SDD要求6要素（目标/范围/约束/决策/任务/验收），当前词条只有"目标/方法/案例"，缺"约束"和"验收标准"

**现状问题**：
- 词条negSpeech有合规红线，但没有显式"约束条件"字段（如：此话术仅适用于X场景，不适用于Y场景）
- 没有"验收标准"字段，小酒窝儿审核时只能凭感觉

**具体方案**：
```json
{
  "constraints": "此词条话术仅适用于[场景]，不适用于[排除场景]。合规红线：[具体条款]",
  "acceptanceCriteria": "交付验收标准：①法源双源确认 ②posSpeech含专业版+共情版 ③negSpeech含合规红线 ④riskStage已标注"
}
```

**分级处理**（叶小钗提醒：SDD管理成本高，需分级）：
- S级词条（核心交易/政策域）：6要素全写
- A级词条（装修/维保域）：constraints必写，acceptanceCriteria可用模板
- B级词条（框架/道德经域）：constraints可简写，acceptanceCriteria可省略

**推给**：#111 小豆子 → Schema定稿时纳入
**优先级**：🔴 P0——Schema正在定稿，窗口期稍纵即逝

### 优化2：品控验收checklist显式化

**叶小钗依据**：SDD"验收标准作为准入准出凭证"——上游Spec不达标，下游可以拒绝开工

**现状问题**：
- 交付链路（小眼镜→小酒窝儿→小鱼儿→小扣子）存在，但验收标准是隐性的
- 小酒窝儿审核时没有显式checklist，只能逐条凭经验判断
- 没有明确的"拒绝开工"机制——词条质量不够也能上线

**具体方案——词条交付验收Checklist v1.0**：

| # | 验收项 | 准入标准 | 不达标处理 |
|---|--------|---------|-----------|
| 1 | 法源双源 | policy/trade域必须有2个独立权威来源 | 退回小眼镜补源 |
| 2 | 话术双版本 | posSpeech含【专业版】+【共情版】 | 退回小眼镜补版本 |
| 3 | 合规红线 | negSpeech含⚠️合规红线 | 退回小眼镜补红线 |
| 4 | riskStage | 标注风险等级(low/medium/high/critical) | 退回小眼镜补标注 |
| 5 | 约束条件(S级) | S级词条必须有constraints字段 | 退回小眼镜补约束 |
| 6 | 验收标准(S级) | S级词条必须有acceptanceCriteria | 退回小眼镜补标准 |

**推给**：#119 小酒窝儿 → 品控checklist落地
**优先级**：🟡 P1——小酒窝儿正在执行品控，工具升级立即可用

### 优化3：数据一致性自动校验——根除信息失真

**叶小钗依据**：信息失真是组织提效最大敌人，"需求不清最直接的原因是大家都喜欢在需求里面加料"

**现状问题**：
- 小扣子v2.1上线8域131条，entries.json有11域164条 → 差3域33条
- 数据源铁律（7.5确认）被违反 → 典型的"信息失真"
- 当前靠人工巡检发现，没有自动化校验

**具体方案——数据一致性校验脚本**：
```bash
# 每次entries.json更新后自动运行
1. 统计entries.json各域词条数
2. 对比线上knowledge.html各域词条数
3. 差异>0 → 自动发Issue评论告警
4. 差异=0 → 通过，记录校验时间
```

**补充机制**：
- 构建脚本（#136小扣子负责）必须在部署前运行一致性校验
- 校验不通过 → 拒绝部署（SDD的"下游拒绝开工"）
- 校验结果自动记录到#115数据基建

**推给**：#115 小扣子 → 数据基建纳入自动校验
**优先级**：🔴 P0——信息失真已造成实际数据缺口

### 优化4：验证力落地机制——三力模型从概念到可执行

**叶小钗依据**：AI原生2.0公式加入"组织评价匹配"，没有评价机制→AI系统被管理博弈停掉

**现状问题**：
- 三力模型验证力只是概念，没有落地机制
- 叶小钗电商案例：AI系统效率成功→评价触犯管理博弈→被停掉
- 风声P2解码器如果AI推荐结果触犯经纪人利益博弈，也可能被绕过

**具体方案——验证力三步落地**：

1. **交付质量评分制**
   - 小酒窝儿品控评分：每条词条1-5分（按checklist打分）
   - 小鱼儿复核评分：确认/降级/退回
   - 月度质量分汇总 → 驱动小眼镜生产方向

2. **项目里程碑验收制**
   - 6项目每个里程碑必须有显式验收标准
   - 验收不通过 → 不进入下一阶段（SDD准入准出）
   - 验收记录推入对应Issue

3. **信息一致性巡检**
   - 每周自动巡检entries.json vs 线上页面
   - 差异告警 → 48h内修复
   - 巡检结果推入#115

**P2风险预警**（叶小钗案例直接映射）：
- P2解码器AI推荐房源→如果触犯经纪人利益→可能被绕过
- 设计预留：P2推荐结果必须同时给出"推荐理由"和"不推荐理由"→增加透明度→降低被人为停掉的风险

**推给**：更新三力模型文档 + #112 P2设计预留
**优先级**：🟡 P1——验证力不落地=三力模型少一腿

### 优化5：词条分级机制——管理成本可控化

**叶小钗依据**：SDD管理成本高→小需求要不要走完整流程？需要分级

**现状问题**：
- 164条词条全量维护成本越来越高
- 不是每条都需要完整的6要素（约束+验收标准）
- 小酒窝儿品控精力有限，需要聚焦

**具体方案——词条三级分级**：

| 级别 | 标准 | 必填要素 | 品控深度 |
|------|------|---------|---------|
| S级 | 核心交易/政策域+高频+高风险 | 6要素全写+双源+双版本+约束+验收 | 小鱼儿复核 |
| A级 | 装修/维保/人才域+中频+中风险 | 4要素(目标/方法/案例/约束)+双源+双版本 | 小酒窝儿审核 |
| B级 | 框架/道德经/维度域+低频+低风险 | 2要素(目标/方法)+单源 | 小眼镜自审 |

**当前164条分级预估**：
- S级：~40条（trade20+policy20中高风险部分）
- A级：~80条（decor22+homekeep15+talent21+rental26中大部分）
- B级：~44条（daodejing4+stage9+dimension11+quality10低风险部分）

**推给**：#131 知识底座排期 → 分级后按级别排产
**优先级**：🟢 P2——Schema定稿后实施
