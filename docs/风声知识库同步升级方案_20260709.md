# 风声知识库同步升级方案

> 来源：坚果云7.9推文「Obsidian新出4种同步策略」+ 生哥确认
> 当前痛点：entries.json每次更新要手动推GitHub + 手动PUT坚果云，两个动作分开做，容易漏
> 目标：三端同步从手动变自动

---

## 一、现状

| 端 | 同步方式 | 问题 |
|----|---------|------|
| GitHub | python脚本调用API push | 需要手动触发，每次写python代码 |
| 坚果云 | WebDAV API手动PUT | 需要先MKCOL创建目录，再PUT上传，两步操作 |
| 本地Obsidian | 无同步 | 知识库md文件只在Agent workspace，Obsidian无法直接访问 |

**每次词条更新流程**：
1. 写entries_merged_XXX.json
2. 跑python脚本推GitHub（commit+push）
3. 跑python脚本PUT坚果云（MKCOL+PUT）
4. 三个动作分开，漏任何一个=三端不一致

---

## 二、4种同步策略解析（坚果云Nutstore Sync插件）

据[坚果云公众号7.9推文](https://mp.weixin.qq.com)，Nutstore Sync插件新增4种同步策略：

| 策略 | 核心逻辑 | 类比 | 适用场景 |
|------|---------|------|---------|
| **Mirror（镜像）** | 本地=云端，双向完全同步 | 照镜子 | 只有一台主力设备，要云端实时备份 |
| **Backup（备份）** | 本地→云端单向，云端改动不回传 | 存钱罐 | 只管备份不管回读，防止云端误操作影响本地 |
| **Archive（归档）** | 本地→云端按版本归档 | 邮局寄信 | 需要保留每次变更的独立版本 |
| **Selective（选择）** | 只同步指定文件夹/文件 | 快递柜 | 只同步部分内容，不同步全量 |

## 三、风声推荐方案

### 方案：Obsidian + Git插件 + Nutstore Sync（Backup模式）

**为什么选Backup模式**：
- 风声知识库只有小眼镜在写，不存在多端编辑冲突
- Backup=本地→云端单向，**云端误删不影响本地**——这比Mirror安全
- 坚果云作为备份端，GitHub作为版本控制端，各司其职

### 架构图

```
小眼镜workspace（数据源）
    │
    ├─→ Obsidian Vault（本地）
    │     ├─ Git插件 → 自动commit+push → GitHub（版本历史）
    │     └─ Nutstore Sync（Backup模式） → 坚果云（实时备份）
    │
    └─→ entries.json → GitHub data/entries.json（线上数据源）
```

### 具体配置

#### 1. Obsidian Vault结构
```
风声知识库/
├── 方法论知识库/          ← 已有内容，同步到坚果云
├── 内容素材/              ← 已有内容，同步到坚果云
├── 知识底座生产/          ← entries.json等，同步到坚果云
├── .gitignore             ← 排除.obsidian等
└── .obsidian/             ← Obsidian配置（不同步）
```

#### 2. Git插件配置
- 插件：Obsidian Git（社区插件市场安装）
- Auto commit-and-sync interval：5分钟
- Auto commit after stopping file edits：开启
- Auto pull interval：5分钟
- .gitignore：排除.obsidian/、.DS_Store、*.tmp

#### 3. Nutstore Sync配置
- 插件：Nutstore Sync（社区插件市场安装）
- 同步策略：**Backup（备份）**
- Remote directory：/风声协作/风声知识库/
- Auto-sync interval：2分钟

#### 4. entries.json特殊处理
- entries.json放在Vault根目录或知识底座生产/子目录
- Git追踪：✅（版本历史+线上数据源）
- 坚果云同步：✅（备份）
- GitHub的data/entries.json通过Git自动同步，不需要单独push

### 优化效果

| 动作 | 升级前 | 升级后 |
|------|--------|--------|
| 推GitHub | 手动写python脚本push | Git插件自动commit+push |
| 推坚果云 | 手动WebDAV PUT | Nutstore Sync自动备份 |
| 三端一致性 | 靠人记得住两步都做 | 两个插件各自自动跑 |
| 版本回溯 | 靠GitHub commit记录 | GitHub+坚果云双重版本历史 |
| 操作成本 | 每次更新5-10分钟 | 0分钟（全自动） |

## 四、风险与应对

| 风险 | 概率 | 应对 |
|------|------|------|
| 坚果云免费版流量限制（每月1GB上传） | 中 | 风声知识库全是md/json，单次几十KB，够用 |
| Git冲突（多端同时编辑） | 低 | 只有Agent写，不存在多端冲突 |
| 插件停更 | 低 | Git插件社区活跃度高；Nutstore是坚果云官方维护 |
| entries.json被云端误删 | 低 | Backup模式单向，云端删不影响本地；GitHub有版本历史 |
| Agent workspace和Obsidian Vault路径不同 | 中 | 需要配置软链接或手动同步一次初始文件 |

## 五、实施步骤

1. **安装Obsidian**（如未安装）
2. **创建Vault**：指向`/app/data/所有对话/主对话/`或创建新Vault后软链接
3. **安装Git插件** + 配置自动commit/push
4. **安装Nutstore Sync插件** + 配置Backup模式
5. **初始化Git仓库**：关联fengsheng-tasks仓库
6. **验证**：修改一个文件 → 等5分钟 → 检查GitHub和坚果云是否同步
7. **退役旧流程**：不再手动python push + WebDAV PUT

## 六、备选方案

如果Obsidian Vault路径和Agent workspace无法直接打通，备选：

**方案B：CodeAct脚本自动化**
- 写一个CodeAct脚本：entries.json更新后自动推GitHub+坚果云
- 一条命令完成三端同步
- 不依赖Obsidian，但不如Obsidian插件那样全自动

**方案C：坚果云客户端文件夹同步**
- 安装坚果云桌面客户端
- 将workspace目录设为同步文件夹
- 客户端自动同步，但不如Git插件有版本控制
