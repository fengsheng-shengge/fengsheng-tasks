# 风声终端记忆层 · 主动触发机制设计

> 核心问题：记忆层从"被拉取"到"被触发"，经纪人没问，系统就知道他现在需要什么
> 对标：Apple Vision Pro的ambient sensing / Google Astra的连续视觉理解 / Meta Ray-Ban的语音上下文
> 小眼镜 · 2026-07-21

---

## 一、问题：当前记忆层是"被动"的

| 当前模式 | 触发方式 | 问题 |
|---------|---------|------|
| 搜索 | 经纪人输入关键词 | 必须先知道要搜什么 |
| 场景卡片 | 经纪人点击首页 | 必须先识别自己在哪个场景 |
| 游戏 | 经纪人主动进入 | 跟工作场景脱节 |
| 词条 | 经纪人搜索后展示 | 只有找到才看得到 |

**这就是"数据库"和"记忆"的根本区别**——数据库是被查的，记忆是自己冒出来的。你跟客户聊到押金，不需要想"我去查一下退租押金怎么算"，你的经验会自动跳出来。AI眼镜要的就是这个。

## 二、行业对标：贝壳VR看房已经验证了"场景触发+内容推送"

贝壳VR看房的逻辑：
- 客户进入VR看房 → 平台自动推送话术模板+房源信息给经纪人
- 本质：**场景触发 + 内容推送**，经纪人不需要自己去找

这个逻辑已经被验证了——经纪人接受、习惯、依赖这种方式。但贝壳只做了一小段：

| 维度 | 贝壳VR看房 | 风声终端记忆层 |
|------|-----------|--------------|
| **触发源** | 客户在APP上的行为（点击/浏览/进入VR） | 经纪人在真实业务中的对话（说话/见客户/签合同） |
| **触发范围** | 只在APP内，经纪人不在线就断 | 真实工作中，说话的每一刻都能触发 |
| **推送内容** | 话术模板+房源信息（帮卖房） | 专业方法论+法律依据+案例佐证（帮做对的事） |
| **内容性质** | 销售支持 | 专业支持 |
| **覆盖场景** | 获客→带看→成交（到签约为止） | 全生命周期（签约前→签约中→签约后→居住中→退租出售） |
| **空白场景** | 签约后的纠纷、退租押金、交房验收（跟佣金无关不碰） | 这些恰恰是经纪人天天遇到、客户最需要专业支持的 |

**结论：贝壳验证了"场景触发+内容推送"的产品形态，但只覆盖了交易场景。风声在专业场景做同样的事，覆盖更广、内容更深、触发方式更自然。**

## 三、头部企业怎么做的

### 2.1 Apple Vision Pro — 空间感知触发
- 持续感知：你在看什么、你在哪、周围有什么
- 空间锚点：走到厨房自动调出菜谱，走到白板前自动识别笔迹
- **关键能力**：视觉上下文理解（你看到的 = 你需要的上下文）

### 2.2 Google Astra — 连续视觉对话
- 手机摄像头持续开启，实时理解画面内容
- 你指着零件问"这个怎么装"，它已经知道你在看什么
- **关键能力**：多模态上下文（视觉+语音+历史对话连续体）

### 2.3 Meta Ray-Ban — 语音上下文触发
- 持续监听语音，识别对话中的关键实体
- 客户说"我担心学区"，眼镜自动推送学区政策
- **关键能力**：语音关键词实时触发

### 2.4 蚂蚁灵波 — 任务状态触发
- 机器人知道自己在执行哪个步骤
- 当前步骤决定下一步调用什么能力
- **关键能力**：工作流状态机驱动

### 2.5 共性提炼

**主动记忆触发的三个要素**：

1. **上下文感知**（你在哪、你在看什么、你在跟谁说话）
2. **意图推断**（你现在大概率需要什么）
3. **实时推送**（不需要你问，主动送到你眼前/耳边）

## 三、风声记忆层的主动触发架构

### 3.1 触发信号层：系统怎么知道经纪人在什么场景

| 信号类型 | 信号源 | 能推断什么 | 当前可否实现 |
|---------|--------|-----------|------------|
| **语音信号** | AI眼镜麦克风 | 客户正在说什么话题、情绪、关键实体 | ✅ 技术成熟 |
| **位置信号** | GPS/室内定位 | 经纪人在哪（售楼处/签约中心/客户家） | ✅ 技术成熟 |
| **视觉信号** | AI眼镜摄像头 | 经纪人在看什么（合同/房源/客户表情） | ⚠️ 需OCR+场景识别 |
| **时间信号** | 日历/任务系统 | 当前处于交易哪个阶段 | ✅ 可对接 |
| **行为信号** | 小程序/APP操作 | 经纪人刚才查了什么、收藏了什么 | ✅ 已有数据 |
| **对话信号** | 语音对话的连续体 | 从上一句推断下一句需要什么 | ⚠️ 需对话状态追踪 |

### 3.2 触发规则层：什么信号→激活什么记忆

**这是整个主动记忆的核心——把"经纪人遇到的场景"翻译成"记忆层该推送什么"。**

#### 规则类型1：关键词触发（最基础，语音/文本信号）

```
触发条件：对话中出现关键词实体
映射规则：关键词 → 场景域 → 记忆包

示例：
  "押金" → rental.退租押金 → 推送SCENE-01退租押金场景卡+法条+案例
  "学区" → trade.学区房 → 推送学区承诺工具卡+案例
  "定金" → trade.定金纠纷 → 推送SCENE-05定金纠纷+定金vs订金对比
  "阴阳合同" → trade.阴阳合同 → 推送SCENE-07+法律红线
  "户口" → trade.户口迁出 → 推送SCENE-03+迁出时限
```

**实现要求**：
- 关键词表从241条词条的consumerQ+search_aliases提取，约400-500个触发词
- 每个触发词映射到1个场景+1-3张工具卡
- 延迟<2秒（从听到到推送到眼镜屏幕/耳机）

#### 规则类型2：场景状态触发（位置+时间信号）

```
触发条件：位置/时间/日历匹配特定场景
映射规则：场景状态 → 预加载记忆包

示例：
  进入签约中心 → 预加载"签约中"场景域全量记忆
  打开合同页面 → 预加载合同审查工具卡+常见陷阱
  约定交房日当天 → 推送交房验收清单
  客户续租到期前30天 → 推送续租/退租决策卡
```

**实现要求**：
- 6个用户场景域各自有一个"预加载包"
- 进入场景后不是搜索，是全量推送+实时筛选
- 位置围栏触发（地理围栏API）

#### 规则类型3：对话流触发（连续对话信号）

```
触发条件：对话状态追踪（Dialog State Tracking）
映射规则：上一轮意图 → 下一轮可能需要

示例：
  客户问"这个房子有没有抵押" → 推送产权查询流程卡
  客户说"我想退租" → 推送退租押金场景卡+违约金计算
  客户沉默/犹豫 → 推送当前话题的clientBenefit（帮经纪人找说服角度）
  客户情绪激动 → 推送安抚话术+案例（"有个客户也遇到类似情况..."）
```

**实现要求**：
- 对话状态机：每个场景域定义5-10个典型对话路径
- 每个路径节点预映射对应记忆
- 客户沉默/情绪变化是高价值触发点——经纪人最需要帮手的时刻

#### 规则类型4：工作流状态触发（任务/日历信号）

```
触发条件：交易流程推进到特定阶段
映射规则：流程节点 → 该节点的专业记忆

签约前流程：
  首次带看 → 房源评估卡+户型优缺点
  复看 → 对比决策卡+价格谈判策略
  出价 → 定金流程卡+违约风险
  签约 → 合同审查清单+常见陷阱

签约后流程：
  付款 → 贷款方案对比+首付红线
  过户 → 产权查询+税费计算
  交房 → 验收清单+常见纠纷
```

**实现要求**：
- 每个交易流程节点预映射3-5张工具卡
- 流程推进时自动切换记忆包，不需要手动切换
- 对接小程序/APP的交易流程状态

### 3.3 记忆推送层：触发后怎么送到经纪人面前

| 终端 | 推送方式 | 交互方式 | 延迟要求 |
|------|---------|---------|---------|
| AI眼镜 | AR悬浮卡片+语音提示 | 眼球确认/语音追问 | <2秒 |
| AI耳机 | 语音播报（一句话结论） | 语音追问细节 | <1秒 |
| 小程序 | 场景卡片自动弹出 | 点击查看详情 | 页面级 |
| APP | 推送通知+场景卡片 | 主动查看 | 秒级 |

**关键原则**：
- **轻推送**：只推一句话结论（oneLineAnswer），不推全文
- **可追问**：经纪人语音追问"展开说一下"才展开详情
- **不打扰**：同一场景5分钟内不重复推送
- **可关闭**：经纪人可以静默模式，只看不推

## 四、记忆层的触发索引设计

当前记忆层只有"词条ID"一个索引，远远不够主动触发。需要建立多维索引：

### 4.1 触发词索引（trigger_words.json）

```json
{
  "triggerWords": [
    {
      "word": "押金",
      "sceneId": "SCENE-01",
      "toolIds": ["RNT-015", "RNT-016"],
      "priority": 1,
      "context": "退租/租赁纠纷"
    },
    {
      "word": "学区",
      "sceneId": "SCENE-06",
      "toolIds": ["TRX-012"],
      "priority": 1,
      "context": "购房/签约"
    }
  ]
}
```

- 从241条词条的consumerQ + 68条search_aliases提取
- 预计400-500个触发词
- 每个触发词映射到场景+工具卡+优先级

### 4.2 场景状态索引（scene_state_map.json）

```json
{
  "sceneStates": [
    {
      "state": "签约中",
      "triggers": ["location:签约中心", "calendar:签约日", "keyword:合同/签字/首付"],
      "preloadToolIds": ["TRX-001", "TRX-005", "TRX-008", "TRX-012"],
      "preloadSceneIds": ["SCENE-05", "SCENE-07"]
    }
  ]
}
```

- 6个用户场景域 × 3-5个状态 = 约20-30个场景状态
- 每个状态定义触发条件+预加载记忆包

### 4.3 对话流索引（dialog_flow_map.json）

```json
{
  "dialogFlows": [
    {
      "sceneId": "SCENE-01",
      "flowId": "退租押金-典型路径",
      "nodes": [
        {
          "nodeId": "退租押金-01",
          "trigger": "客户说想退租",
          "pushMemory": {"toolId": "RNT-015", "focus": "退租通知期+押金退还条件"},
          "nextNodes": ["退租押金-02", "退租押金-03"]
        },
        {
          "nodeId": "退租押金-02",
          "trigger": "房东说扣押金",
          "pushMemory": {"toolId": "RNT-016", "focus": "扣押金合法情形+案例"},
          "nextNodes": ["退租押金-04"]
        }
      ]
    }
  ]
}
```

- 每个场景域3-5条典型对话路径
- 每条路径5-10个节点
- 每个节点定义触发条件+推送记忆

### 4.4 工作流状态索引（workflow_state_map.json）

```json
{
  "workflows": [
    {
      "workflowId": "trade-purchase",
      "name": "二手房购买流程",
      "stages": [
        {
          "stageId": "带看",
          "triggerCalendar": "带看预约",
          "triggerLocation": "房源地址",
          "preloadToolIds": ["TRX-003", "TRX-006"],
          "dialogFlowId": "trade-带看-典型路径"
        }
      ]
    }
  ]
}
```

- 租赁流程（5-6个阶段）+ 交易流程（6-8个阶段）
- 每个阶段定义日历/位置触发+预加载记忆

## 五、实现路径

### Phase 1 · 关键词触发（P0-P1期间，7-8月）
- **产出**：trigger_words.json，400-500个触发词
- **技术要求**：语音识别+关键词匹配，延迟<2秒
- **对接终端**：小程序搜索增强（"客户问了你什么"入口的智能推荐）
- **验证**：50个高频卡点场景中，40个以上能被触发词命中

### Phase 2 · 场景状态+对话流触发（P2期间，8-9月）
- **产出**：scene_state_map.json + dialog_flow_map.json
- **技术要求**：对话状态追踪+上下文理解
- **对接终端**：小程序场景自动切换 + AI耳机语音推送
- **验证**：6个场景域各跑通1条典型对话路径，经纪人无需手动搜索

### Phase 3 · 工作流状态+全量触发（P3期间，9-10月）
- **产出**：workflow_state_map.json + 全量四索引联动
- **技术要求**：位置围栏+日历集成+工作流状态机
- **对接终端**：AI眼镜全量接入
- **验证**：完整交易流程（签约→过户→交房）全程自动推送，经纪人不主动搜索也能得到专业支持

## 六、跟AI眼镜合作的接口标准

当风声的记忆层要被AI眼镜调用时，眼镜厂商需要的不是我们的数据，而是：

1. **触发规则包**：什么信号→激活什么记忆（trigger_words + scene_state_map）
2. **记忆内容包**：激活后推送什么内容（oneLineAnswer + 可追问详情）
3. **对话流包**：客户说了A，经纪人接下来可能需要什么（dialog_flow_map）
4. **反馈接口**：经纪人是否采纳了推送、追问了什么、跳过了什么→ 用于优化触发精度

**接口设计**：

```
POST /memory/trigger
请求：{
  "signals": [
    {"type": "keyword", "value": "押金"},
    {"type": "location", "value": "签约中心"},
    {"type": "dialog", "value": "客户说想退租但房东不肯退押金"}
  ],
  "context": {
    "workflow_stage": "退租",
    "previous_queries": ["退租通知期"]
  }
}

响应：{
  "triggered": true,
  "sceneId": "SCENE-01",
  "push": {
    "headline": "退租押金：房东不是想扣就能扣",
    "oneLineAnswer": "正常磨损不算损坏，房东无权扣押金",
    "confidence": 0.92,
    "drilldown": {
      "toolIds": ["RNT-015", "RNT-016"],
      "caseIds": ["CASE-RNT-001", "CASE-RNT-005"],
      "legalBasis": ["《民法典》第714条", "《民法典》第715条"]
    }
  },
  "followUp": [
    "客户问扣多少合理？→ RNT-015",
    "房东说有损坏？→ CASE-RNT-001"
  ]
}
```

**这个接口就是AI眼镜厂商接入风声记忆层的唯一入口——他们只需要调这个API，不用管我们内部怎么分类、怎么索引、怎么联动。**

## 七、用户价值排序：哪些触发方式经纪人真的会用

从经纪人的核心工作场景出发，他的动作就三个：**说话、看东西、按流程走**。

| 触发方式 | 用户价值 | 原因 | 优先级 |
|---------|---------|------|--------|
| **语音触发** | ★★★★★ | 80%时间在跟客户说话，零操作成本，说话即触发 | **P0** |
| **对话流触发** | ★★★★★ | 客户说了一个处境，系统理解后推送专业依据，帮经纪人"接话" | **P0** |
| **流程节点触发** | ★★★★ | 签约→付款→过户→交房，每步该用什么确定性强，不需要AI猜 | **P1** |
| 搜索触发 | ★★★ | 确定性最强，但依赖主动发起，不是增量 | 已有 |
| 位置触发 | ★★ | 依赖硬件，场景有限（签约中心/售楼处），锦上添花 | P2 |
| 视觉触发 | ★★ | 依赖AI眼镜摄像头+OCR，技术成本高，场景不成熟 | P3 |

**核心判断：语音触发+对话流触发 = 记忆层从"被动数据库"变"主动记忆"的核心方向。**

经纪人不会为了触发记忆去买AI眼镜，但会因为"说话就能用"而离不开。先把这两个做透，跟AI眼镜合作时就是现成的护城河能力。

## 八、核心判断

**从"数据库"到"记忆"的质变，不是数据量的问题，是触发机制的问题。**

- 数据库：你问，我答
- 记忆：你还没问，我已经知道你需要什么

跟AI眼镜合作的壁垒不是241条词条，是400个触发词+20个场景状态+30条对话流+12个工作流节点——这些才是让眼镜"主动"而不是"被动"的关键。

**搭记忆层的数据是苦力活，搭触发机制才是护城河。**

---

## 九、核心壁垒：三层焊死，行业无人做到

| 层级 | 能力 | 行业现状 |
|------|------|---------|
| **感知层** | 能触发——知道你现在需要什么 | 语音助手/眼镜厂商能感知，但没有行业专业记忆，触发了也给不了专业内容 |
| **判断层** | 给提示——告诉你该注意什么 | 法律AI能给提示，但得你主动去查，不能场景触发 |
| **行动层** | 给方法——告诉你怎么做 | 工具卡不是知识片段，是"核心洞察+法律依据+行动指令+案例佐证"的可执行方案 |

**三层叠在一起才是完整的：能触发 + 给提示 + 给方法 = 专业大脑，不是知识库。**

为什么AI眼镜厂商需要风声：他们有感知能力（语音/视觉），但垂直行业的专业记忆层是空的，自己搭不出来。风声的记忆层不是数据，是"感知→判断→行动"的闭环能力。

**这就是行业壁垒：三层焊死，目前没有人做到。**
