# AI团队共享记忆方法论提炼

> 来源：公众号"Vayne-LV manisfast"《AI团队最大的浪费：验证过的经验，没有变成团队资产》
> 原文发布：2026-07-06
> 提炼入库：2026-07-12 小眼镜

---

## 一、核心观点

**被验证过的经验，才是AI Native团队最值钱的资产。**

AI团队最大的浪费不是token，不是算力，而是：验证过的经验没有被下一次复用。

## 二、关键框架

### 2.1 团队记忆 = 被压缩过的工程经验

团队记忆不是聊天记录的堆砌，而是经过验证、压缩、结构化的工程经验。

| 维度 | 聊天记录 | 团队记忆 |
|------|---------|---------|
| 结构 | 碎片化对话 | 结构化文档 |
| 验证 | 未验证 | 已验证可用 |
| 可检索 | 难检索 | 秒级检索 |
| 可复用 | 不可复用 | 下次直接调取 |
| 更新机制 | 无 | 每次验证后更新 |

### 2.2 工作记忆 vs 长期记忆

- **工作记忆**：当前任务的上下文（对话历史、当前文件、短期目标）
- **长期记忆**：被验证过的工程经验（架构决策、踩坑记录、最佳实践、模板）

流转逻辑：
1. 执行任务 → 产生工作记忆
2. 验证通过 → 压缩为长期记忆
3. 下次执行 → 自动调取长期记忆
4. 发现过时 → 更新长期记忆

### 2.3 Git+Markdown作为记忆仓库

- Git：版本控制 + 变更可追溯
- Markdown：人类可读 + AI可读 + 工具可解析
- 多Agent共享：Claude/Cursor/Codex/Trae读写同一仓库

### 3.2 缺的（缺口30%）

| 文章要素 | 风声缺口 | 优先级 | 行动 |
|---------|---------|--------|------|
| 工作记忆→长期记忆压缩协议 | 词条生产完后的经验没有自动回流到执行协议 | P1 | 话术三关通过后的"经验摘要"字段 |
| 执行时自动调取相关经验 | memory_search是语义检索，但没有"任务类型→经验模板"的映射 | P1 | 建立任务类型→经验索引 |

| Agent间经验传递协议 | Issue评论是传递通道，但没有标准化的"经验传递格式" | P3 | 定义经验传递模板 |

### 3.3 关键差距：经验没有回流到执行

**现状**：词条生产→三关通过→推GitHub+坚果云→结束

**应该**：词条生产→三关通过→提取经验摘要→回流到执行协议→下次同类任务自动调取

例子：
- 我处理了GitHub Contents API 404问题→找到Git Data API绕过→这个经验应该回流到TOOLS.md
- 我发现Python heredoc不能用特殊Unicode→这个经验已回流到TOOLS.md ✅
- 但"severity判定规则"只在我脑子里，没有结构化回流到可自动调取的位置

## 四、风声行动项

### A1. Schema加lastVerified字段（P1）
- 在Schema v1.0中增加`lastVerified: "YYYY-MM-DD"`字段
- 每次词条被实际使用/验证后更新此字段
- 过期词条（>90天未验证）自动标记为"待复验"

### A2. 经验回流协议（P1）
- 话术三关通过后，自动生成"经验摘要"（该词条生产过程中的关键决策点）
- 经验摘要写入TOOLS.md锚点行 + experience/详情文件
- 已在做但不够系统化，需要标准化流程

### A3. 任务类型→经验索引（P1）
- 按任务类型（词条生产/脚本部署/方案撰写/Issue跟踪）建立经验索引
- 每次执行前自动检索相关经验
- 目前靠memory_search，需要更精确的映射

### A4. 知识资产面板（P2）
- 管理看板加知识库健康度卡片：词条数/三关通过率/最后验证日期分布/域覆盖度
- 数字员工考核指标已出，待对接看板

### A5. 经验传递模板（P3）
- Agent间传递经验时使用标准格式：背景→问题→验证方案→结论→适用条件
- 优先在Issue评论中试点

## 五、与7.8生哥指令对照

生哥7.8说"你要帮我验证他们是按你说的执行的"→ 这篇文章说"验证过的经验才是资产"

**本质上是一回事**：
- 生哥要的"验证他们执行" = 工作记忆→长期记忆的验证环节
- 文章说的"经验变资产" = 验证后压缩回流

风声已经有的三关验证机制，比文章说的还严格（severity+双版本+合规红线三关），但**验证后没有回流到执行协议**这一步是缺口。
