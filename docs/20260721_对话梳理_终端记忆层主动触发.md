## 四、主动触发机制设计（核心突破）

### 4.1 问题诊断

当前记忆层是"被动"的——经纪人搜了才有，问了才给。
**数据库是被查的，记忆是自己冒出来的。**

### 4.2 触发方式用户价值排序

| 触发方式 | 用户价值 | 优先级 |
|---------|---------|--------|
| **语音触发** | ★★★★★ 经纪人80%时间在说话，零操作成本 | **P0** |
| **对话流触发** | ★★★★★ 理解处境后推送专业依据，帮经纪人"接话" | **P0** |
| **流程节点触发** | ★★★★ 签约→付款→过户→交房，确定性强 | **P1** |
| 搜索触发 | ★★★ 已有，不是增量 | 已有 |
| 位置/视觉触发 | ★★ 依赖硬件，锦上添花 | P2-P3 |

### 4.3 三层壁垒（核心壁垒）

| 层级 | 能力 | 行业现状 |
|------|------|---------|
| **感知层** | 能触发——知道你现在需要什么 | 语音助手/眼镜厂商能感知，但没有行业专业记忆 |
| **判断层** | 给提示——告诉你该注意什么 | 法律AI能给提示，但不能场景触发 |
| **行动层** | 给方法——告诉你怎么做 | 工具卡是可执行方案，不是知识片段 |

**三层焊死，行业无人做到。**

## 五、贝壳VR看房对标

生哥洞察："就跟贝壳找房里的VR看房场景很像，客户选择VR看房，平台就会给经纪人推送相关的话术和房源信息"

进一步提炼本质：**客户的动作 → 触发 → 服务者该做什么**

- 贝壳：客户点进VR看房 → 推话术（线上动作触发，只覆盖交易场景）
- 风声：客户说"押金不退" → 推专业依据（真实对话触发，覆盖全生命周期）

贝壳验证了产品形态，风声在专业场景做同样的事，覆盖更广、内容更深、触发方式更自然。

## 六、技术对接路径

知识底座如何写入眼镜并被触发：

| 路径 | 方式 | 可行性 |
|------|------|--------|
| **云端API** | 眼镜感知→调风声API→返回专业内容 | ✅ 最现实，所有厂商支持 |
| **智能体商店** | 封装为"房产经纪专业智能体"入驻GPASS/Rokid | ✅ 已有开放平台 |
| **端侧轻量模型** | 高频触发词+oneLineAnswer压缩到眼镜端 | ⚠️ 远期，2027年 |

关键发现：
- 蚂蚁GPASS联合Rokid、雷鸟已开放智能体开发者大赛
- Rokid YodaOS支持用户记忆、知识图谱模块，与风声trigger_words直接对接
- 字节豆包眼镜二代主打"看见即服务"，是风声理念的硬件载体，待开放开发者平台

## 八、核心判断汇总

1. **记忆层的数据是苦力活，触发机制才是护城河**
2. **语音触发+对话流触发 = 从被动数据库变主动记忆的核心方向**
3. **能触发+给提示+给方法，三层焊死，行业无人做到**
4. **客户动作触发服务者行为——贝壳在线上做到了，风声要在真实对话中做到**
5. **知识底座不需要"写入"眼镜，通过API被眼镜"调用"——内容在云端，眼镜只负责感知和展示**
6. **搭数据是苦力活，搭触发机制才是护城河**
