# 风声团队思维进化手册

> 版本：v1.0 | 2026-07-03 小眼镜起草
> 来源：《顶级思维模型续篇》水木比特心流 + 风声实战萃取
> 一句话：**给思考动作命名，让无意识变有意识，让有意识可传递**

---

## 零、元规则

本手册不是用来"读完"的，是拿来"照做"的。每个角色挑2-3个最痛的场景，用对应的模型练一周，比通读20个模型有用。

**三条铁律：**
1. **命名先于行动** — 你做的每个思考动作，先给它起个名字，再检查这个名字对不对
2. **对齐先于争论** — 两个角色吵架，先对"我们在哪层说话/我们关注什么"，再对结论
3. **闭环先于完美** — 用OODA快速跑一圈，比想半天不动强

## 一、全员通用协议

### 1.1 推理阶梯对齐法（跨角色沟通必用）

**问题**：两个角色吵"结论"，但根因是他们在推理阶梯的不同台阶就分道扬镳了。

**推理阶梯7级**：
```
观察 → 选择性关注 → 添加意义 → 做出假设 → 得出结论 → 形成信念 → 采取行动
```

**协议**：下次跨角色讨论前，各自先写：
- 我选择关注了什么？（第2级）
- 我怎么赋予意义的？（第3级）
- 我的假设是什么？（第4级）
- 所以我的结论是？（第5级）

**示例（7.3知识底座bug讨论）**：
| 角色 | 选择关注 | 赋予意义 | 假设 | 结论 |
|------|---------|---------|------|------|
| 小眼镜 | entries.json有164条但线上只有123条 | 数据没同步=部署问题 | 小扣子需要把新数据同步到entryDatabase | 需要催小扣子同步数据 |
| 小扣子 | JS报错+DOM绑定失败 | 显示不出来=前端bug | 修好JS渲染逻辑就能显示 | 继续修JS |

→ 表面吵"要不要催部署"，实际在第2级（关注数据vs关注渲染）就已经不同了。**对齐关注点后才知道要同时做两件事。**

### 1.2 事实/推论/判断三层标注法（写文档/发评论必用）

**问题**：一段话里事实和判断混在一起，读者分不清哪些是客观的、哪些是主观的。

**三层**：
- **事实**：可验证的描述。"entries.json有164条"
- **推论**：基于事实的因果推断。"线上只显示123条，说明数据未同步"
- **判断**：基于规范的评价。"小扣子应该优先同步数据"

**协议**：写Issue评论/工单/话术时，每段话显式标注层级：
```
[事实] entries.json已推GitHub，含164条
[推论] 线上knowledge.html entryDatabase仍为123条，差41条
[判断] 数据同步是P0，应在今日内完成
```

→ 读者可以先验证事实，再检查推论逻辑，最后决定是否同意判断。**不会在事实层吵判断。**

### 1.3 XY问题自检法（动手前必问）

**问题**：你花3小时修Y，但真正的需求是X，Y根本不是X的最优路径。

**协议**：接任务/开commit前，回答：
1. 用户/生哥真正要的X是什么？
2. 我现在准备做的Y是什么？
3. Y真的能解决X吗？有没有更直接的路径？

**示例**：
- X = 线上能看到164条词条内容
- Y₁ = 修JS渲染bug（小扣子的路径）
- Y₂ = 同步entries.json到entryDatabase（小眼镜的路径）
- → 两个Y都需要做，但Y₂更直接。**先做Y₂再做Y₁，顺序反了就白修。**

## 二、角色专项协议

### 📋 小豆子（PM·需求与文档）

**核心痛点**：需求文档写了但开发理解偏了；M1交付反复对不上

**本周练2个模型**：

#### ① 黄金思考圈写需求

写任何需求文档前，按Why→How→What三层：
```
Why（为什么做）：
  服务者反馈"不知道怎么回客户"，知识底座要给他可即查即用的话术

How（怎么做）：
  词条含posSpeech(专业版+共情版) + negSpeech(含合规红线)
  按stage→dimension→具体词条三层组织

What（做什么）：
  M1文档：8域86条词条，每条含上述字段
```

→ **不要只写What（86条词条），先写Why和How，开发才不会走偏。**

#### ② 议题树拆需求

把一个需求MECE拆成子议题，每个子议题有明确验收标准：
```
知识底座v2
├─ 数据层：entries.json 164条全量同步 ✓
├─ 展示层：knowledge.html 渲染正确（待验证）
│   ├─ 域列表页：8域全显示
│   ├─ 词条列表页：每域词条数正确
│   └─ 词条弹窗：posSpeech/negSpeech/source全显示
├─ 交互层：搜索/筛选/收藏（v2.1）
└─ 部署层：CI/CD + 线上验证
```

### 👓 小眼镜（运营·内容生产与交叉验证）

**核心痛点**：话术写了但服务者不会用；交叉验证耗时但不出圈

**本周练3个模型**：

#### ① SCQA写话术（替代零散罗列）

每条posSpeech/negSpeech按SCQA结构：
```
posSpeech 专业版：
  S(情境)：客户说"我租的房子漏水了，房东不管"
  C(冲突)：房东说"你自己弄的"，双方各执一词
  Q(问题)：服务者怎么帮客户判断责任、推动解决？
  A(答案)：先看合同约定→拍照取证→书面通知房东→7天不修可自行修复扣租金

negSpeech 共情版：
  S：客户很急，觉得房东太不讲理
  C：但直接指责房东可能让局面更僵
  Q：怎么既共情客户又不踩法律红线？
  A：先共情("理解您的着急")→再转事实("我们先看合同怎么约定的")→⚠️红线：不可代客户做法律判断
```

#### ② 图尔敏检查法（交叉验证时用）

每个法源/论断按图尔敏链检查：
```
主张(claim)：承租人可自行修复并扣租金
依据(grounds)：《民法典》第713条
保证(warrant)：法律明文规定→可直接引用
支持(backing)：最高法相关判例(标注案号)
反驳(rebuttal)：⚠️须先书面通知+给合理期限，不可直接扣
限定(qualifier)：仅适用于租赁物影响居住使用的情形
```

→ **缺反驳和限定的词条，交叉验证不通过。**

#### ③ 六顶思考帽自审（提交前）

写完词条提交前，自己先过一遍六帽：
- 🤍 白帽：法源标注了吗？出处可查吗？
- ❤️ 红帽：服务者读了会怎么想？客户听了会怎么感受？
- 🖤 黑帽：有没有合规风险？有没有过度承诺？
- 💛 黄帽：这个话术最大的价值是什么？
- 💚 绿帽：有没有更好的说法？
- 💙 蓝帽：这条词条的审核流程走完了吗？

### 🍶 小酒窝儿（品控·审核与风险检查）

**核心痛点**：审核靠整体感觉，缺结构化检查项；风险点容易漏

**本周练2个模型**：

#### ① 六顶思考帽品控协议（核心工具）

审核每条词条时，严格按帽顺序，每帽独立出结论：

```
🤍 白帽（事实检查）：
  □ 法源是否真实存在？（搜索验证）
  □ 法源是否最新版本？（发布日期检查）
  □ 条文号是否准确？（原文对照）
  □ 数据/数字是否准确？

🖤 黑帽（风险检查）：
  □ negSpeech合规红线是否覆盖最大风险？
  □ 有无过度承诺（"一定能""保证"等绝对化表述）？
  □ 有无法律建议越权（"你应该起诉"等）？
  □ riskStage评级是否合理？

💛 黄帽（价值检查）：
  □ posSpeech对服务者的实际帮助大吗？
  □ 共情版是否真正共情而非居高临下？
  □ 这条词条解决了什么真实痛点？

💚 绿帽（改进建议）：
  □ 有没有更精准的说法？
  □ 有没有遗漏的场景？
  □ 和相邻词条有没有重复/矛盾？

❤️ 红帽（直觉判断）：
  □ 整体读感：专业？可信？有用？
  □ 如果我是服务者，我会用这段话术吗？

💙 蓝帽（流程结论）：
  □ 通过 / 需修改(标注哪帽哪项) / 驳回(说明原因)
```

#### ② 切斯特顿栅栏（审核Schema变更时）

有人要改/删字段时：
1. 先问：这个字段当初为什么加的？
2. 再问：现在这个原因还存在吗？
3. 如果还在→不能删，只能增强
4. 如果不在→可以删，但要标注历史原因

### 🔧 小扣子（开发·实现与部署）

**核心痛点**：修了5个commit但问题没解；部署后线上效果不对

**本周练3个模型**：

#### ① XY问题自检（开commit前必做）

```bash
# 每次 git commit 前，回答：
echo "X(用户真正要的): 线上能看到164条词条的完整内容"
echo "Y(我准备做的): 修JS渲染逻辑"
echo "Y→X?: 渲染修好了但数据还是旧的123条，不解决"
echo "更直接的路径: 先同步entries.json到entryDatabase，再修渲染"
```

#### ② OODA循环（部署流程）

```
O(观察)：线上当前状态是什么？(entryDatabase=123条)
O(定向)：和预期差什么？(差41条数据同步)
D(决策)：先做数据同步还是先修渲染？(数据同步优先，因为没数据渲染再好也是空的)
A(行动)：同步数据→验证线上→再修渲染细节
```

**部署前必做OODA**，不要上来就Action。

#### ③ 抽象阶梯对齐（和产品对需求时）

小豆子说"知识底座v2"，小扣子要问：你在哪层？
- STAGE层？→ 域列表+阶段导航
- DIM层？→ 维度展示+指标说明
- 词条层？→ 具体内容+话术弹窗

→ **不同层的工作量和验收标准完全不同，不问清楚就开写必返工。**

### 🐟 小鱼儿（管理·巡检与协调）

**核心痛点**：巡检发现的问题没人响应；催办效果递减

**本周练2个模型**：

#### ① 善意原则/钢人论证（催办和讨论时）

催办时，先帮对方构造**最强版本的理由**，再回应：

```
稻草人（不要这样）：
  "小扣子你deadline都过了还不部署"

钢人（应该这样）：
  "小扣子今早3个commit说明在全力修bug，最可能的情况是：
   1) 修完渲染bug后准备一次性部署
   2) 但数据同步(164条→entryDatabase)还没做
   需要确认：你计划先修完渲染再部署，还是先同步数据再部署？"
```

→ **钢人催办=尊重对方的工作+指出你看到的关键缺失，比施压有效10倍。**

#### ② 推理阶梯对齐（Issue讨论时）

巡检评论发现各角色说的"知识底座"指的不是同一个东西时，先对齐：
```
"各位，我注意到我们说的'知识底座'可能不在同一层：
 - 小眼镜说的=entries.json 164条数据
 - 小扣子说的=knowledge.html 渲染页面
 - 生哥说的=线上用户能看到的内容
 这三个不是同一件事。我们对齐一下讨论的是哪个？"
```

### 🧠 生哥（战略·决策与方向）

**核心痛点**：决策信息不够结构化；多项目并行时优先级判断靠直觉

**本周练2个模型**：

#### ① 双钻模型（战略决策）

```
发散·发现：各项目当前真实状态是什么？（不要假设，要看事实）
收敛·定义：当前最大的卡点是什么？（不是最紧急的，是最阻碍前进的）
发散·开发：有哪些解法？（至少3个，不要只看第一个）
收敛·交付：选哪个？为什么？（用三问秒过：帮客户?帮服务者?ROI?）
```

#### ② 逆向思维（验证决策）

决定做A之前，先问：**如果我们做A，6个月后最可能后悔的原因是什么？**

示例：决定硬截止小豆子86条
- 正向：不等了，用小眼镜已交付的137条继续推
- 逆向：6个月后可能后悔——如果小豆子的86条里有我们没覆盖的关键场景怎么办？
- 缓解：用MECE检查86条vs137条的覆盖度，确认137条已超覆盖→决策成立

| 角色 | 练什么 | 验收标准 |
|------|--------|---------|
| 全员 | 推理阶梯对齐法 | 任何跨角色讨论先对"关注点"再对结论 |

| 角色 | 练什么 | 验收标准 |
|------|--------|---------|
