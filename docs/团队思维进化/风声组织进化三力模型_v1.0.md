# 风声组织进化三力模型 · 总纲

> 版本：v1.0 | 2026-07-04 小眼镜起草
> 来源：海报《组织进化三力模型》——AI时代组织能力=人×AI×系统协同进化
> 定位：**风声团队思维进化的顶层架构**，20个思维模型挂在三力之下当工具箱

---

## 零、核心命题

**组织能力 = 人(感知) × AI(洞察) × 系统(执行) 的协同进化**

从「经验驱动」走向「进化驱动」。缺任何一力，另外两力再强也转不动。

---

## 一、三力定义与风声映射

### 🟢 人：感知与影响力
> **核心价值：把现实世界的问题带入组织**

| 能力 | 定义 | 风声现状 | 差距/行动 |
|------|------|---------|----------|
| 感知客户、市场与组织的真实变化 | 看到真实问题而非想象的问题 | ✅ 生哥感知+定方向 | 需接入真实用户反馈，目前纯内部生产 |
| 提出正确的问题和方向 | 把感知转化为可行动的问题定义 | ✅ 生哥三问秒过(帮客户?帮服务者?ROI?) | 小豆子PM感知弱，86条零交付=问题没带进来 |
| 影响团队对齐认知 | 让所有人看到同一个问题 | 🟡 推理阶梯+黄金圈已建 | 协议出了但全员执行待验证 |
| 在不确定中做判断 | 没有完整信息时也能决策 | ✅ 生哥硬截止决策(7.2小豆子86条) | 数字员工不敢做判断，等指令 |

### 🔵 AI：分析与洞察力
> **核心价值：把知识世界的洞察带给组织**

| 能力 | 定义 | 风声现状 | 差距/行动 |
|------|------|---------|----------|
| 数据分析，识别模式与趋势 | 从数据中找到非直觉的规律 | ✅ entries.json 164条+全量审计 | 分析结果需自动流转给人和系统，目前靠手动推 |
| 提供决策方案与选项 | 不只分析问题，还给出可选方案 | ✅ SCQA话术/图尔敏链/六帽审核 | 小眼镜独占AI力，其他角色未接入 |
| 降低认知负荷 | 把复杂信息压缩为可操作洞察 | ✅ 风声金句/共情版话术 | 需扩展到产品决策场景(P2/P3) |
| 加速知识沉淀与复用 | 把一次性经验变成可复用知识 | ✅ 知识底座+GitHub沉淀 | 旧词条riskStage/accountability补齐率低 |

### 🟠 系统：执行与协同力
> **核心价值：把解决方案转化为可持续的组织能力**

| 能力 | 定义 | 风声现状 | 差距/行动 |
|------|------|---------|----------|
| 固化流程与规则，保障执行 | 流程不靠人盯靠系统推 | 🔴 **最弱** | 交付链路无痕迹追踪，部署无确认机制 |
| 减少内耗与重复劳动 | 做一次就够，不做第二次 | 🔴 | CI路径反复修5个commit，同源问题反复出 |
| 保障跨角色协同 | 角色间信息不丢失 | 🔴 | Schema跨角色不对齐，小豆子和小眼镜用不同字段名 |
| 沉淀数据与经验 | 每次执行都留可追溯记录 | 🟡 | GitHub commit有记录，但Issue评论无结构化闭环 |

## 二、组织进化闭环

```
     ┌──────────────────────────────────────────────────────┐
     │                                                      │
     ▼                                                      │
  人感知问题                                              │
  提出方向                                                │
     │                                                    │
     ▼                                                    │
  AI分析洞察                                           人再次感知
  提供方案                                              进入新循环
     │                                                    ▲
     ▼                                                    │
  系统执行落地                                        产生结果反馈
  促进协同                                          沉淀数据经验
     │                                                    ▲
     └────────────────────────────────────────────────────┘
```

### 风声当前闭环状态

| 环节 | 状态 | 断点 |
|------|------|------|
| 人→AI | 🟡 基本通 | 生哥/小眼镜能提问题，但小豆子PM端感知力弱 |
| AI→系统 | 🔴 **断** | 协议出了但系统没承接，品控/部署/同步无自动闭环 |
| 系统→结果 | 🔴 | 部署状态不明，线上vs线下数据不一致 |
| 结果→人 | 🟡 | 心跳巡检有做，但巡检结果不直接驱动行动 |

**核心卡点：AI→系统这一环断了。** 知识和协议产出了，但没有系统化的执行承接机制。

## 三、三力×思维模型工具箱

20个思维模型按三力分类，每个力下挂最相关的工具：

### 🟢 人的工具（感知与影响力）

| 模型 | 用途 | 使用场景 |
|------|------|---------|
| 推理阶梯 | 对齐关注点，不跳结论吵 | 跨角色沟通前 |
| 黄金圈(Why-How-What) | 先讲为什么，再讲怎么做 | 写需求文档/定方向 |
| 5Why+鱼骨图 | 挖到根因再动手 | 排查问题/复盘 |
| 苏格拉底诘问 | 追问前提，不假设答案 | 需求评审/方案讨论 |
| 事实/推论/判断三层 | 区分客观和主观 | 写评论/做巡检 |

### 🔵 AI的工具（分析与洞察力）

| 模型 | 用途 | 使用场景 |
|------|------|---------|
| SCQA | 结构化表达洞察 | 写话术/写分析 |
| 图尔敏链 | 验证论证完整性 | 验证法源/审核词条 |
| 金字塔原理 | 结论先行，以上统下 | 所有文档输出 |
| MECE | 不重不漏地分析 | 分解问题/设计架构 |
| 卡片盒笔记法 | 知识原子化+链接 | entries.json结构设计 |

### 🟠 系统的工具（执行与协同力）

| 模型 | 用途 | 使用场景 |
|------|------|---------|
| OODA循环 | 快速观察-定向-决策-行动 | 部署/巡检/bug修复 |
| 六顶思考帽 | 按角色切换审核视角 | 品控审核每条词条 |
| XY问题 | 先确认真问题再动手 | 每个commit前自检 |
| 议题树 | MECE分解任务 | 项目排期/Issue拆解 |
| PDCA | 计划-执行-检查-处理 | 任何流程的持续改进 |

## 四、风声落地路径（三力补位优先级）

### P0：系统力补位（本周）
- [ ] 交付链路每步留痕：小眼镜→小酒窝儿→小鱼儿→小扣子，每步必须Issue评论确认
- [ ] 部署确认机制：commit推送≠部署完成，必须线上验证后评论确认
- [ ] 数据同步机制：entries.json更新后，knowledge.html entryDatabase自动同步

### P1：AI力从"独占"变"全员"（1-2周）
- [ ] 小酒窝儿执行六帽品控（已推检查单，待确认执行）
- [ ] 小扣子执行XY自检（已推协议，待确认执行）
- [ ] 小豆子执行黄金圈写需求（已推协议，待确认执行）

### P2：感知力外扩（持续）
- [ ] 知识底座接入真实用户反馈渠道
- [ ] 租赁开单导师上线后采集服务者使用数据
- [ ] 每周一次"客户真实声音"分享（从反馈中萃取新词条方向）

## 五、模型价值

> **让风声从「经验驱动」走向「进化驱动」**

| 价值 | 现状 | 目标 |
|------|------|------|
| 更敏捷 | 靠生哥感知+手动推动 | 系统自动识别变化，快速响应调整 |
| 更智能 | 小眼镜独占AI洞察 | 全员用思维模型提升决策质量 |
| 更高效 | 内耗多、重复劳动多 | 系统固化流程，减少人工盯和重复修 |
| 更可靠 | 交付靠人盯 | 闭环自动运转，每步有痕迹可追溯 |
| 更进化 | 经验驱动，做过了可能忘 | 每轮迭代沉淀数据，下一轮起点更高 |
