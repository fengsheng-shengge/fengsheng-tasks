## 一、质量规范总纲

### 1.1 三道铁律（不变）

| 铁律 | 标准 | 一票否决 |
|------|------|---------|
| 找得到 | 至少2个入口可达（搜索/场景/法条/案例/游戏） | 不入库 |
| 看得懂 | 3秒测试：标题3秒懂/结论30字内/动作明确 | 不入库 |
| 用得上 | brokerNote含可执行动作指令，不是知识科普 | 不入库 |

### 1.2 两层质量标准（新增）

5000+词条必须分两层管理，一层保广度，一层保深度：

**标准层（词典词条，约4500条）**

| 字段 | 要求 | 说明 |
|------|------|------|
| id | 域前缀+序号 | 如 TRD-001、POL-050 |
| domain | 所属数据域 | 签约前/签约中/签约后/居住中/退租出售/职业成长 |
| subScene | 子场景 | 如"签约前→资格审查→限购" |
| name | 词条名（客户问法） | 必须是客户能问出来的问题，不是专业术语 |
| alias | 搜索别名5-8个 | 口语化+近义词+场景词 |
| def | 专业定义 80-200字 | 一句话说清楚这是什么、为什么重要 |
| corePoint | 核心要点 3-5条 | 每条≤20字，经纪人能直接转述 |
| legalRef | 法律依据 | 引用到条文号，如"《民法典》第710条" |
| consumerQ | 客户典型问法 | 经纪人在客户面前最常被问到的问题 |
| oneLineAnswer | 一句话答案 ≤30字 | 经纪人能3秒转述给客户 |
| consumerBenefit | 客户受益点 | 告诉客户"这对你有什么好处" |
| severity | hard/medium/soft | hard=法条强制/medium=标准规范/soft=行业惯例 |
| lastVerified | 时效验证日期 | 格式YYYY-MM-DD，超期标黄 |
| relatedEntries | 关联词条ID | 至少2条，形成知识网络 |
| source | 来源 | 法条/政策/标准/行业惯例 |

**深度层（核心词条，约500条）**

在标准层全部字段基础上，增加：

| 字段 | 要求 | 说明 |
|------|------|------|
| brokerNote | 经纪人行动指令 | 包含明确动词："告诉客户/提醒客户/建议客户" |
| posSpeech | 正向话术（专业版+共情版） | 在客户面前怎么说 |
| negSpeech | 负向话术+合规红线 | 不能怎么说+法律边界 |
| caseIds | 关联案例1-2条 | hard词条必须2条（1正1负） |
| toolType | 工具卡类型 | 决策卡/流程卡/评估卡/校准卡/风险卡/边界卡 |
| lawLayer | 规律锚定 | 市场规律/法律规律/人性规律至少一层 |
| sceneId | 所属场景卡片 | 从场景入口可达 |

### 1.3 severity分级标准（细化）

| 级别 | 定义 | 标准层占比 | 深度层占比 | 示例 |
|------|------|-----------|-----------|------|
| hard | 法条强制，违规有法律后果 | 30% | 50% | 押金退还标准、阴阳合同违法 |
| medium | 标准/规范/政策，不遵守有风险 | 40% | 35% | 网签流程、公积金贷款条件 |
| soft | 行业惯例/经验判断，做对了更好 | 30% | 15% | 看房时间选择、议价策略 |

### 1.4 数据源权威分级（不变）

| 等级 | 来源 | 标注要求 |
|------|------|---------|
| A | 法律条文/司法解释/国务院法规/住建部规章/地方法规 | 标注发布机关+文号+日期，可直接引用 |
| B | 行业标准/协会规范/官媒报道 | 标注"需交叉验证" |
| C | 自媒体/营销号/个人经验 | 不收录 |

### 1.5 入库三关（强化版）

```
第1关：找得到吗？
  → 别名≥5个？
  → 至少2个入口可达？
  → 触发词覆盖？（进小程序"客户说了什么"能命中？）

第2关：看得懂吗？
  → name是客户问法？
  → oneLineAnswer≤30字？
  → corePoint每条≤20字？
  → 非专业人员3秒能判断"这说的是什么"？

第3关：用得上吗？
  → 有法律依据可引用？
  → 有客户受益点？
  → 深度层：brokerNote有明确动作？
  → 深度层：有案例佐证？

三关全过→入库。任何一关不过→打回修改，不过不占数。
```

### 1.6 四层联动标准（扩量版）

5000+词条下，联动关系必须自动化维护：

| 联动关系 | 标准层 | 深度层 |
|---------|--------|--------|
| 词条→关联词条 | ≥2条 | ≥3条 |
| 词条→场景卡片 | 至少1个场景可达 | 直接挂载场景卡 |
| 词条→案例 | 不强制 | hard≥2条(1正1负)，medium≥1条 |
| 词条→工具卡 | 不强制 | 1:1映射 |
| 词条→触发词 | ≥3个触发词 | ≥5个触发词 |

### 1.7 时效验证标准

| 词条类型 | 核查周期 | 超期处理 |
|---------|---------|---------|
| 法条类(hard) | 每季度 | 超90天标黄"待验证"，超180天标红"可能过期" |
| 政策类(medium) | 每半年 | 超180天标黄 |
| 惯例类(soft) | 每年 | 超365天标灰"需确认" |

## 二、生产体系

### 2.1 三条产线并行

```
产线A：AI批量生产（标准层主力）
  法条/政策原文 → AI解读初稿 → 自动化校验 → 人工抽检 → 入库
  日产能：80-120条标准层词条
  质量守门：人工抽检率20%，抽检不过整批打回

产线B：人工精写（深度层主力）
  高频卡点场景 → 小眼镜精写 → 7字段完整 → 小酒窝儿审核 → 入库
  日产能：10-15条深度层词条
  质量守门：100%人工审核

产线C：经纪人UGC（长尾补充）
  经纪人纠错/验证/补充 → 自动初筛 → 人工终审 → 入库
  日产能：上线后自然增长，预计日均5-10条
  质量守门：自动初筛+人工终审
```

### 2.2 AI批量生产流程（产线A详细）

```
Step 1：数据源采集
  → 裁判文书网/法律法规库/住建部官网/地方住建局
  → 每批次采集50-100条原始法条/政策

Step 2：AI解读初稿
  → 输入：法条原文+所属场景域
  → 输出：name/alias(5-8个)/def/corePoint/legalRef/consumerQ/oneLineAnswer/consumerBenefit/severity
  → 要求：name必须是客户问法，oneLineAnswer≤30字

Step 3：自动化校验
  → 字段完整性：必填字段是否齐全
  → 字数校验：oneLineAnswer≤30字，corePoint每条≤20字
  → 重复检测：name/alias是否与已有词条重复
  → 法条格式：legalRef是否标注到条文号
  → 别名唯一性：alias是否与其他词条的alias冲突

Step 4：人工抽检
  → 抽检率20%
  → 抽检项：name是否客户问法/def是否准确/oneLineAnswer是否可转述/legalRef是否正确
  → 抽检通过率≥95%→整批入库
  → 抽检通过率<95%→整批打回，调整prompt重跑

Step 5：入库+关联
  → 自动生成relatedEntries（基于domain+subScene+legalRef同源）
  → 自动生成触发词（从alias+consumerQ提取）
  → 写入entries_5000.json
```

### 2.3 人工精写流程（产线B详细）

```
Step 1：选题
  → 从高频卡点场景清单选题
  → 优先级：hard>medium>soft，签约前>签约中>其他

Step 2：精写
  → 按7字段深度层标准写
  → brokerNote必须含动作指令
  → posSpeech/negSpeech各两个版本（专业版+共情版）
  → caseIds至少1条，hard≥2条

Step 3：自检
  → 过入库三关
  → 过3秒测试
  → 链路完整性：词条→工具卡→案例→法条无断裂

Step 4：审核
  → 小酒窝儿品控审核
  → 审核项：专业准确性/法条正确性/案例真实性/话术可用性

Step 5：入库
  → 写入entries_5000.json + broker_toolkit.json + case_library.json
  → 同步更新场景卡片/搜索别名/触发词
```

| 租客纠纷应对 | 押金纠纷/租客违约/转租/群租/宠物/噪音/损坏赔偿 | "租客把房子弄坏了押金够赔吗""群租违法吗怎么处理" | ~50 |

**业主侧词条的核心定位：不是给业主看的，是经纪人服务业主时的专业弹药。**

经纪人服务的"客户"=消费者（买方/租客）+ 业主（卖方/房东）。业主侧词条解决的是：
- 经纪人帮业主卖房时，能不能说出"独家委托3天无买家可解约"的法律依据
- 经纪人帮业主出租时，能不能告诉业主"租客拖欠租金满15天可解除合同"
- 业主问"税费谁承担"时，经纪人能不能当场给出政策依据
- 业主问"租客把房子弄坏了"时，经纪人能不能给出维权路径

**这就是帮服务者配得上被业主好好对待——业主也是客户。**

**业主侧词条的special requirements：**
- severity分布：hard占比更高（50%），因为业主面对的权属/税务问题法律后果更重
- 双Q字段：保留consumerQ（消费者问法，经纪人帮消费者时用）+ 新增ownerQ（业主问法，经纪人帮业主时用）
- 双Card字段：保留clientCard（面向消费者的话术）+ 新增ownerCard（面向业主的话术）
  - ownerCard结构：{title, oneLineAnswer, ownerBenefit}
  - ownerBenefit：告诉业主"这样做对你有什么好处"（保住资产/降低风险/合法维权）
- 案例方向：偏权属纠纷+税务判例+租赁纠纷判例
- 正向案例占比更高（业主更需要"做对了→保住资产"的信心，而不是被吓唬）
- brokerNote需同时覆盖：经纪人如何帮消费者应对业主 + 经纪人如何帮业主应对消费者（双视角）

**同一词条的双视角示例：**

| 词条 | consumerQ（消费者问经纪人） | ownerQ（业主问经纪人） |
|------|---------------------------|----------------------|
| 押金退还 | "房东扣我押金说墙面磨损" | "租客把墙弄花了押金能扣吗" |
| 提前退租 | "我要提前退租房东不退押金" | "租客突然要退租我损失谁赔" |
| 独家委托 | — | "中介让我签独家委托，该签吗" |
| 房产税 | — | "我两套房房产税怎么算" |
| 租客违约 | — | "租客拖欠租金3个月了怎么办" |
