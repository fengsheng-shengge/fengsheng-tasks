# 数字员工协同进化·自闭环路谱方案

> **目标**：数字员工形成自闭环组织——任务从需求→设计→开发→品控→部署→验证全链路自动流转，组织记忆持续积累，协同效率指数级提升
> **起点**：2026-07-21 交接文档标准化完成
> **终态**：L5自闭环组织（日常迭代零人工调度）

---

| 维度 | 现状 | 问题 |

| 自动化 | 仅脚本级（定时任务） | 任务级零自动化 |

## 二、目标状态：L5自闭环组织

```
需求进入 → 自动识别类型 → 自动分派 → 自动流转 → 自动品控 → 自动部署 → 自动验证
                                                        ↑                    |
                                                        └── 不通过自动打回 ←─┘
                                                        
组织记忆：每次协同的经验自动沉淀 → 下次更高效
异常处理：自动重试→失败升级→通知小鱼儿→人工介入
```

**L5定义**：日常迭代（如小程序V2.x）零人工调度，小鱼儿只做3件事：
1. 方向决策（做什么不做什么）
2. 异常升级处理（自动流转失败时）
3. 组织进化（优化协同规则本身）

### Level 2 → Level 3：工单流转标准化

**核心机制**：每个任务有标准工单，工单在员工间自动流转

#### 工单生命周期

```
创建(需求) → 设计(小豆子) → 开发(小扣子) → 品控(小酒窝) → 部署(小鱼儿) → 验证 → 关闭
              ↓              ↓              ↓              ↓
           设计文档       开发+交接文档   审核报告       部署确认
```

#### 工单标准格式

```json
{
  "id": "FS-2026-07-001",
  "type": "feature|bugfix|content|data",
  "priority": "P0|P1|P2",
  "status": "draft|designing|developing|reviewing|deploying|verifying|closed|rejected",
  "title": "首页触发入口",
  "creator": "生哥",
  "assignee": "小豆子",
  "history": [
    {"from": "draft", "to": "designing", "by": "小鱼儿", "at": "2026-07-20T10:00"},
    {"from": "designing", "to": "developing", "by": "小豆子", "at": "2026-07-20T14:00", "artifact": "设计文档路径"}
  ],
  "artifacts": {
    "design_doc": "路径",
    "handover_doc": "路径",
    "review_report": "路径"
  },
  "auto_rules": {
    "review_pass": "auto_deploy",
    "review_fail": "auto_return_to_dev",
    "deploy_fail": "notify_xiaoyuer"
  }
}
```

#### 流转规则（核心）

| 当前状态 | 完成动作 | 下一状态 | 执行者 | 触发条件 |
|---------|---------|---------|--------|---------|
| draft | 确认需求 | designing | 小豆子 | 优先级+描述齐全 |
| designing | 交付设计文档 | developing | 小扣子 | 设计文档通过格式校验 |
| developing | 交付代码+交接文档 | reviewing | 小酒窝 | 构建通过+交接文档6板块齐全 |
| reviewing | 审核通过 | deploying | 小鱼儿 | 审核清单全打勾 |
| reviewing | 审核不通过 | developing | 小扣子 | 附修改意见自动打回 |
| deploying | 部署成功 | verifying | 自动 | 部署后检查项通过 |
| verifying | 验证通过 | closed | 自动 | 线上验证通过 |
| verifying | 验证失败 | deploying | 小鱼儿 | 回滚+排查 |

### Level 3 → Level 4：自动调度+品控闭环

**核心机制**：任务进来自动识别类型→自动分派→自动流转，品控通过自动部署

#### 自动分派规则

```yaml
task_routing:
  feature:
    design: 小豆子
    develop: 小扣子
    review: 小酒窝
    deploy: 小鱼儿
  
  content:
    design: 小眼镜
    develop: 小眼镜
    review: 小酒窝
    deploy: 小鱼儿
  
  bugfix:
    design: null  # 跳过设计，直接开发
    develop: 小扣子
    review: 小酒窝
    deploy: 小鱼儿
  
  data:
    design: null
    develop: 小鱼儿
    review: 小酒窝
    deploy: 小鱼儿
```

#### 品控闭环自动化

1. **小酒窝审核完** → 自动判断通过/不通过
2. 通过 → 自动触发部署脚本（小鱼儿的CI/CD）
3. 不通过 → 自动打回小扣子，附审核意见，工单状态回developing
4. 部署完 → 自动跑验证脚本（冒烟测试）
5. 验证通过 → 工单自动关闭+汇报生哥
6. 验证失败 → 自动回滚+通知小鱼儿

#### 组织记忆沉淀

每次流转自动记录：
- **耗时**：每个环节实际耗时 vs 基准
- **打回率**：design→dev打回几次，review打回几次
- **踩坑**：什么坑、谁踩的、怎么解决的
- **模式**：同类任务的典型路径和耗时

→ 沉入 `组织记忆/协同经验/` → 下次同类任务自动参考

### Level 4 → Level 5：自闭环+自适应

**核心机制**：组织根据数据自动优化自身

#### 自适应优化

1. **瓶颈检测**：某环节耗时>基准2倍 → 自动告警+建议拆分/加人
2. **打回率优化**：某类任务review打回率>30% → 自动建议加强设计环节
3. **替补自动触发**：某人超时未交付 → 按替补矩阵自动转交
4. **经验预加载**：新工单创建时 → 自动匹配历史同类工单 → 预加载踩坑记录给执行者

#### 小鱼儿角色进化

```
L2（现在）：调度员——所有事经我手

L4：监护者——规则自动运转，我处理异常
L5：进化者——我优化规则本身，组织自我进化
```

## 五、关键设计决策

### 1. 工单存哪？
**决策**：GitHub Issues + Labels
- 原因：已有仓库，零新增基础设施；Labels做状态机；Comments做流转记录
- 工单号 = Issue号；状态 = Label（designing/developing/reviewing/deploying）
- 交接文档附件 = Issue Comment里的链接

### 2. 流转触发方式？
**决策**：Webhook + 脚本
- GitHub Issue状态变更 → Webhook → 脚本判断→通知下一环节执行者
- MVP阶段：手动Label变更+脚本检测，不做全自动化
- V2阶段：Webhook全自动

### 3. 品控标准怎么定？
**决策**：继承交接文档模板的审核清单
- 每类任务有标准审核清单（feature/bugfix/content各一份）
- 清单全打勾=通过，有一项不勾=打回
- 清单持续迭代（每打回一次，补充一个检查项）

### 4. 组织记忆存哪？
**决策**：GitHub docs/协同经验/ + 本地recent_memory/
- 全局共享：GitHub docs/（跨员工可读）
- 个体经验：各员工自己的记忆文件
- 双写：沉淀时同时写两边
