# 风声终端记忆层 · 知识库标准与验证体系

> 核心原则：记忆层的验证标准不是"我们产出了多少条"，是"经纪人能不能找到、看懂、用上"
> 小眼镜 · 2026-07-21

---

## 一、知识库搭建标准

### 1.1 分类体系：从数据分类→用户分类

**问题**：当前11个域（policy/decor/trade/homekeep/rental/talent/dimension/stage/daodejing/quality-customer/quality-server）是数据视角的分类，经纪人不会这么想。

**标准**：建立双分类体系——数据域（开发用）+ 用户场景域（经纪人用），两层映射。

**用户场景域（经纪人视角）**：

| 一级场景 | 二级场景 | 覆盖数据域 |
|---------|---------|-----------|
| 签约前 | 选房/看房/定价/资质审查 | trade+rental+dimension |
| 签约中 | 合同审查/定金/首付/贷款 | trade+rental+policy |
| 签约后 | 交房/过户/户口/装修 | trade+homekeep+decor+policy |
| 居住中 | 维修/物业/邻里/续租 | homekeep+rental |
| 退租/出售 | 押金/违约/挂牌/出售 | rental+trade+policy |
| 职业成长 | 人才画像/专业度/服务标准 | talent+quality-server+quality-customer+stage |

**每个二级场景就是经纪人脑海中的"文件夹"——他不用知道数据域，只需要知道"我现在在帮客户签合同，给我签约中要用的一切"。**

---

### 1.2 目录结构：三层导航

```
场景入口（客户卡在哪？）
  └── 场景详情页（我该怎么接？）
        ├── 专业判断（工作记忆-工具卡）
        │     ├── 关键点
        │     ├── 法律依据
        │     └── 经纪人使用说明
        ├── 别人踩过的坑/做对的事（经验记忆-案例）
        │     ├── 正向：做对了→好结果
        │     └── 负向：缺失什么→出了问题
        └── 深度知识（长期记忆-词条）
              ├── 客户问法（consumerQ）
              ├── 一句话答案（simpleAnswer）
              └── 客户受益点（consumerBenefit）
```

**标准**：从任何一层进入，都能向上/向下导航到完整链路。不能有"死胡同"——用户看到案例，必须能跳到对应的工具卡和词条；用户看到词条，必须能跳到场景和案例。

---

### 1.3 相关性：显性关联+隐性推荐

**显性关联（硬连接）**：
- 场景卡片 ↔ 工具卡（3-5张）
- 工具卡 ↔ 词条（1条）
- 工具卡 ↔ 案例（1-2条）
- 案例 ↔ 法条（在词条legalSources中）

**隐性推荐（软连接）**：
- **同场景推荐**：同一场景卡片下的其他工具卡
- **对比推荐**：容易混淆的词条（定金vs订金、退租vs违约）
- **进阶推荐**：用户看了基础词条后，推荐相关的hard词条
- **场景链推荐**：签约中→签约后，场景之间的顺承关系

**标准**：每张工具卡至少2条显性关联 + 2条隐性推荐，不能让用户"看完就走"。

---

### 1.4 可检索性：多入口+多路径

**入口1：场景导航**（首页卡片→场景详情→工具卡→词条）
- 用户心智："客户卡在这里了，我该怎么办？"
- 覆盖：8张场景卡片

**入口2：自然语言搜索**（"客户问了你什么？"）
- 用户心智："客户刚说了xxx，我该怎么回？"
- 覆盖：68条搜索别名 + consumerQ关键词
- 标准：搜索命中率≥90%（50个高频卡点场景中，至少45个能通过口语化搜索命中）

**入口3：法条检索**（按法律条文查）
- 用户心智："客户提到了这个法条，相关内容有哪些？"
- 覆盖：legalSources中出现的所有法条
- 标准：每条法条能反向查到所有引用它的词条和案例

**入口4：案例检索**（按判决/场景/正负向）
- 用户心智："有没有类似的真实案例？"
- 覆盖：案例库全量
- 标准：支持按场景/法条/正负向/关键词四种筛选

**入口5：游戏化入口**（客户模拟器→场景→工具卡）
- 用户心智："考考我"
- 覆盖：客户模拟器20场景
- 标准：每道题答完推荐对应工具卡

**总标准**：5个入口，任何一个入口进入都能走完全链路。不允许存在"只能通过一种方式找到"的孤岛内容。

---

### 1.5 内容质量标准

**每条内容必须通过"3秒测试"**：
1. **3秒能看懂标题**：标题必须是客户能问出来的问题，不是专业术语
2. **3秒能理解结论**：oneLineAnswer不超过30字，经纪人能直接转述给客户
3. **3秒能知道下一步**：brokerNote给出明确动作指令，不是知识科普

**过不了3秒测试的内容，不是记忆，是资料库——能查但不能用。**

## 二、验证标准体系

### 2.1 结构验证：四层联动完整性

| 验证项 | 标准 | 检查方法 |
|--------|------|---------|
| 场景→工具卡 | 每张场景卡片关联3-5张工具卡 | 遍历sceneDetails[].tools，检查toolId在broker_toolkit中存在 |
| 工具卡→词条 | 每张工具卡映射1条词条 | 遍历toolkit[].id，检查在entries.json中存在 |
| 工具卡→案例 | 每张工具卡至少1条案例，hard的至少2条（1正1负） | 遍历toolkit[].relatedCaseIds，检查在case_library中存在 |
| 案例→法条 | 每条案例引用的法条在词条legalSources中存在 | 遍历cases[].relatedLawIds，检查对应词条的legalSources |
| 搜索→场景→工具卡→词条 | 每条搜索别名能走完全链路 | 遍历search_aliases，验证链路无断裂 |
| 死胡同检查 | 不存在"只能通过一种方式找到"的内容 | 从每个入口遍历，检查是否存在不可达节点 |

### 2.2 可用性验证：经纪人能不能用起来

| 验证项 | 标准 | 检查方法 |
|--------|------|---------|
| 3秒标题测试 | 标题是客户能问出来的问题 | 抽样20条，非专业人员3秒内能判断"这说的是什么" |
| 3秒结论测试 | oneLineAnswer≤30字 | 程序检查所有toolkit[].clientCard.oneLineAnswer字数 |
| 3秒动作测试 | brokerNote给出明确动作 | 人工检查：是否包含"告诉客户/提醒客户/建议客户"等动词 |
| 搜索命中率 | 高频卡点场景搜索命中≥90% | 取50个高频场景，口语化搜索，统计命中数 |
| 跨入口可达 | 每条内容至少2个入口可达 | 遍历全量内容，统计每个内容的可达入口数 |

### 2.3 质量验证：内容本身够不够硬

| 验证项 | 标准 | 检查方法 |
|--------|------|---------|
| 法条可追溯 | hard词条的法条标注到条文号 | 检查legalSources格式（如"《民法典》第725条"） |
| 案例可验证 | A级案例有法院/官媒原始URL | 检查case[].sourceUrl可访问 |
| 正负均衡 | 正向案例≥30% | 统计direction=positive占比 |
| 佐证强度 | hard词条佐证强度≥中 | 检查severity=hard的evidenceStrength |
| 分类分布 | 6类工具卡每类≥5张 | 统计toolType分布 |

## 三、每个交付物的验收检查单

### 场景卡片验收
- [ ] painPoint是客户的原话/口语化表达
- [ ] professionalTake是经纪人能直接转述的判断
- [ ] legalBasis标注到条文号
- [ ] cases至少2条，正负至少各1条
- [ ] tools关联的工具卡在toolkit中存在
- [ ] 从搜索别名能命中这张卡片

### 工具卡验收
- [ ] 标题是客户问法（不是专业术语）
- [ ] oneLineAnswer≤30字
- [ ] keyPoints 3-5个，每个是一句可执行判断
- [ ] hard词条有legalBasis
- [ ] relatedCaseIds至少1条（hard的2条）
- [ ] lawLayer至少锚定一层规律
- [ ] brokerNote包含明确动作指令
- [ ] 对应词条在entries.json中存在

### 案例验收
- [ ] 有法院/官媒原始URL（A级）
- [ ] judgment是判决结论，不是故事
- [ ] direction标注正向/负向
- [ ] professionalInsight是经纪人可执行的动作
- [ ] 引用的法条在词条legalSources中存在
- [ ] 来源等级标注（A/B，C级不收录）

### 搜索别名验收
- [ ] 每个词条5-6个别名
- [ ] 别名包含：口语化问题+近义词+关键词+场景词
- [ ] 别名能命中对应的场景卡片或工具卡
- [ ] 无重复别名（不同词条不能共享同个别名）

## 四、持续验证机制

### 自动化检查（CodeAct脚本，每日运行）
1. **链路完整性扫描**：遍历全量数据，检查四层联动无断裂
2. **字段完整性检查**：每条内容的必填字段是否齐全
3. **死胡同检测**：是否存在只有一个入口可达的内容
4. **字数检查**：oneLineAnswer是否超30字
5. **案例URL可访问性**：A级案例的sourceUrl是否仍可访问

### 人工抽检（小酒窝儿品控）
1. **3秒测试**：抽样20条，非专业人员验证
2. **专业准确性**：法条引用是否正确
3. **场景覆盖率**：高频卡点清单中的场景是否都有内容

### 用户验证（上线后）
1. **搜索命中率**：从实际搜索日志统计
2. **使用深度**：用户进入后是否继续浏览关联内容
3. **纠错率**：积分体系驱动的纠错频率

---

## 五、核心判断

**记忆层的成功标准不是"我们有241条词条"，是"经纪人在30秒内找到他需要的专业内容，且内容能直接用"。**

按知识库的标准搭，不是按数据表的标准搭。区别在于：
- 数据表关心：字段完整、条数够、格式对
- 知识库关心：找得到、看得懂、用得上、有关联

每一条新内容入库前，过三关：
1. **找得到吗？**——至少2个入口可达
2. **看得懂吗？**——过3秒测试
3. **用得上吗？**——brokerNote有明确动作

过不了就不入库，宁可少一条，不要一条废物占记忆空间。
