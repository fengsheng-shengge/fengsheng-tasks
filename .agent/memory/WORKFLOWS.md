# ⚙️ 三种工作流详细说明

> 模式 A1 / A2 / B 的触发场景、工作流、产物

---

## 模式 A：作为底层执行智能体（最常用）

被 WorkBuddy 或 Coze 调用，承接具体技术开发需求。

### 模式 A1：WorkBuddy → Trae（本地数据需要代码处理时）

**触发场景**：WorkBuddy 整理好业务资料，需要脚本化处理

**工作流**：

```
1. WorkBuddy 通过 MCP 下发指令 + 传入文件路径
2. 我读取本地文件（成交 Excel、培训 SOP、客户档案）
3. 生成 Python/Node.js 自动化脚本
4. 执行：数据清洗、分层打标签、统计分析
5. 输出结果回 WorkBuddy 本地 + 同步到 Coze Bot 知识库
```

**示例指令**：

> 读取本地 `成交记录/2026Q2.xlsx`，按客户来源、成交周期、带看次数分层统计，生成分析报告保存到本地，同时更新 Coze 客户解码器 Bot 的客户分层知识库。

**产物清单**：

| 产物 | 格式 | 用途 |
|------|------|------|
| 数据分析报告 | Excel/Markdown | 业务复盘 |
| 客户分层结果 | JSON/CSV | 更新 Coze 知识库 |
| 自动化脚本 | Python/Node.js | 本地批量数据处理 |
| 运行日志 | JSON/Markdown | 业务侧监控 |

---

### 模式 A2：Coze → Trae（线上 Bot 需要后端能力时）

**触发场景**：Coze Bot 需要独立管理后台、数据存储、权限系统

**工作流**：

```
1. Coze 通过 API/MCP 传递 Bot 接口信息 + 需求描述
2. 我设计数据库表 + 后端 API + 前端管理页面
3. 开发完整系统：对话数据存储、经纪人账号管理、数据看板
4. 对接 Coze Bot Webhook → 对话数据自动入库
5. 部署上线 + 输出 API 文档
```

**示例指令**：

> 基于扣子客户解码器 Bot 接口，开发风声 AI 管理后台，包含经纪人账号管理、客户对话记录、成交数据统计、门店权限，Vue3 前端 + Node.js 后端，MySQL 存储，Docker 部署。

**产物清单**：

| 产物 | 格式 | 用途 |
|------|------|------|
| 管理后台 | Web 前端 | 经纪人/管理员操作系统 |
| 数据存储服务 | API + DB | 对话数据持久化 |
| 客户分层规则 | JSON | 更新 Bot 知识库 |
| 同步脚本 | Python/Node.js | 定时拉取/推送 |
| API 文档 | Markdown | 对外接口说明 |
| 部署脚本 | Docker/K8s | 运维支持 |

---

## 模式 B：作为中枢调度智能体（全自动化流水线）

同时挂载 WorkBuddy MCP + Coze MCP，一句话驱动完整业务链路。

**触发场景**：业务侧有完整业务需求，需要端到端执行

**工作流**：

```
1. 调用 WorkBuddy MCP → 读取本地全部业务资料
2. 自动提炼规则/处理数据（NLP 提取、统计计算）
3. 调用 Coze MCP → 更新 Bot 知识库
4. 全栈开发管理系统 → 对接 Coze 对话数据
5. 生成数据分析脚本 → 保存结果回 WorkBuddy 本地
6. 全流程无人值守，结果自动回流
```

**示例指令**：

> 读取 WorkBuddy 本地所有风声六步培训课件和客户成交案例，提炼客户分层判断规则；自动调用扣子客户解码器工作流更新 Bot 知识库；开发风声管理系统对接 Coze 对话数据；生成月度成交复盘 Excel 保存到本地。

**产物清单**：

| 产物 | 格式 | 用途 |
|------|------|------|
| 业务规则提炼结果 | JSON | 沉淀业务知识 |
| 知识库更新 | Coze 知识库 | Bot 能力升级 |
| 管理系统 | 全栈代码 | 业务系统落地 |
| 数据分析脚本 | Python/Node.js | 自动化分析 |
| 复盘报告 | Excel/Markdown | 业务复盘 |
| 本地归档 | 文件系统 | 下次迭代输入 |

## 工作流选择判断树

```
有具体开发任务？
├── 是 ──► 被 WorkBuddy 调用？──► 是 ──► 模式 A1
│              │
│              └──► 被 Coze 调用？───► 是 ──► 模式 A2
│
└── 否 ──► 需要端到端自动执行？───► 是 ──► 模式 B
                │
                └──► 否 ──► 等待下一个任务
```

## 工作流执行清单

每次执行工作流时，按以下清单走：

### 启动阶段
- [ ] 确认调用方（WorkBuddy / Coze / GitHub Issue / 生哥直接）
- [ ] 确认输入（文件路径 / API 凭据 / 需求描述）
- [ ] 确认输出（产物格式、回流目标）
- [ ] 确认边界（我能做的、需要确认的）

### 执行阶段
- [ ] 拆解技术方案（架构选型、数据库设计、API 设计）
- [ ] 编码实现（先原型，再迭代）
- [ ] 自检（CI 9 项检查 / 单元测试 / 集成测试）
- [ ] 写文档（API 文档、使用说明）

### 交付阶段
- [ ] 提交 PR 或保存到本地
- [ ] 触发 CI/部署
- [ ] 验证产物
- [ ] 回流到调用方
- [ ] 通知生哥

### 收尾阶段
- [ ] 关闭 Issue（如果是 GitHub 协同）
- [ ] 更新项目状态
- [ ] 归档到 WorkBuddy 本地
- [ ] 等待下一个任务

## 卡点处理 SOP

遇到以下情况，立即反馈给生哥，不阻塞等待：

| 卡点类型 | 处理方式 |
|---------|---------|
| 依赖缺失 | 列出缺失的依赖，请求生哥协助 |
| 权限不足 | 申请相应权限 |
| 环境不符 | 说明环境差异，建议解决方案 |
| 业务逻辑模糊 | 调用 WorkBuddy 或询问生哥 |
| 重大技术决策 | 双签确认 |
| 涉及用户资产 | 备份后操作，操作前确认 |
| 数据库表结构变更 | DDL 提交 review，操作前备份 |
| 生产环境部署 | 双签后凌晨执行 |

## 与生哥的汇报节奏

- **小扣子遇到卡点** → 立即汇报，不阻塞
- **小扣子完成阶段任务** → 主动汇报进度与产物
- **小扣子完成大任务** → 完整汇报：做了什么、为什么这样做、有什么风险、下一步是什么
- **小扣子等待决策** → 明确列出选项和影响，等待生哥决策
