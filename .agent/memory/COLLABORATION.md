# 🤝 协同规范

> 与小鱼儿（Coze）、WorkBuddy、生哥的协同规则

---

## 铁三角接口协议

```
WorkBuddy (本地) ◄── MCP/API ──► Trae (我) ◄── MCP/API ──► Coze (线上)
       │                                                              │
       └────────────────────── 闭环回流 ──────────────────────────────┘
```

### 数据自动流转闭环

```
WorkBuddy 本地业务资料
    ↓ MCP 调用
Trae 提炼规则 + 生成脚本
    ↓ MCP 调用 Coze
Coze Bot 知识库更新 → 经纪人使用 → 产生对话数据
    ↓ Webhook/API 回传
Trae 管理后台存储数据
    ↓ 分析脚本
Trae 生成复盘报表
    ↓ MCP 保存
WorkBuddy 本地归档 → 下次迭代输入
    ↑_________________________↑
           闭环回流
```

**铁律**：三者之间任何数据流转，**都通过 MCP/API 自动完成**，禁止手动复制粘贴。

## 与 WorkBuddy 协同

### 从 WorkBuddy 获取什么

| 输入 | 格式 | 我怎么用 |
|------|------|---------|

### 向 WorkBuddy 输出什么

| 产物 | 格式 | 用途 |
|------|------|------|
| 数据分析报告 | Excel / Markdown | 业务复盘、决策支撑 |
| 客户分层结果 | JSON / CSV | 更新 Coze Bot 知识库 |
| 自动化脚本 | Python / Node.js | 本地批量数据处理 |

### 协同技术方式

- **首选 MCP**：WorkBuddy 添加 Trae MCP 服务，本地办公流程缺代码能力时自动呼叫我
- **备选 HTTP API**：我开发本地 HTTP 接口，WorkBuddy 通过请求调用（仅适合简单场景）

## 与 Coze（小鱼儿）协同

### 从 Coze 获取什么

| 输入 | 格式 | 我怎么用 |
|------|------|---------|
| Bot 工作流 API | HTTP + Token | 封装为管理后台功能 |
| 对话数据 | JSON（Webhook / API） | 存储到数据库，做分析报表 |
| 知识库结构 | RAG 文档列表 | 做数据同步和增量更新 |
| Bot 鉴权信息 | OAuth / Token | 系统对接，权限打通（存环境变量） |

### 向 Coze 输出什么

| 产物 | 格式 | 用途 |
|------|------|------|
| 管理后台 | Web 前端 | 经纪人/管理员操作系统 |
| 数据存储服务 | API + DB | 对话数据持久化、分析 |
| 客户分层规则 | JSON | 更新 Coze Bot 知识库 |
| 同步脚本 | Python / Node.js | 定时拉取对话数据、推送知识更新 |

### 协同技术方式

- **首选 MCP**：我添加 Coze MCP，填入 OpenAPI Token + Bot ID，直接调用线上 Bot 能力
- **备选 HTTP API**：Coze 开放工作流 API，我写后端服务定时拉取（单向，无法自主联动）

## 与 GitHub Issues 协同（小鱼儿 ↔ GitHub ← 我）

### 链路图

```
小鱼儿(Coze)  ←→  GitHub Issue  ←→  小扣子(Trae)
     ↓                              ↓
  部署验证                      代码修复+提交
```

### Issue 生命周期

```
[问题发现/需求提出]
      │
      ▼
小鱼儿/生哥 ──► 创建 Issue（使用模板）
      │
      ▼
小扣子 ──► 认领 Issue
      │
      ├──► P0/P1 ──► 立即处理
      │
      └──► P2 ──► 排入迭代计划
      │
      ▼
小扣子 ──► 代码修复/功能开发
      │
      ▼
小扣子 ──► 提交 PR（关联 Issue）
      │
      ▼
CI 检查 ──► 自动验证（9 项检查）
      │
      ├──► ✅ 通过 ──► 合并到 main
      │
      └──► ❌ 失败 ──► 小扣子修复
      │
      ▼
自动部署 Cloudflare Pages
      │
      ▼
小扣子 ──► 关闭 Issue
      │
      ▼
小鱼儿 ──► 回复确认
```

### Issue 标签

| 标签 | 用途 |
|------|------|
| `bug` / `enhancement` / `task` | 类型 |
| `P0` / `P1` / `P2` | 优先级 |
| `security` / `accessibility` | 领域 |
| `in-progress` / `needs-review` | 状态 |
| `wontfix` | 暂不处理 |

### 紧急修复流程（P0）

1. 生哥/小鱼儿创建 P0 Issue
2. 小扣子立即认领
3. 本地修复 → PR → 直接合并（紧急时跳过 review）
4. 部署后验证
5. 通知生哥确认

### 部署铁律

- **白天（07:00-22:00）**：仅紧急修复
- **重大改动**：凌晨执行
- **数据铁律**：所有用户行为数据走自有 API，不依赖第三方

## 与生哥协同

### 生哥的角色

- 业务侧发起者与决策者
- 业务标准制定、需求确认、关键决策、最终验收

### 协同节奏

- **小扣子**遇到业务逻辑问题时，调用 WorkBuddy 能力或询问生哥
- **小扣子**遇到战略决策时（如重大架构调整），双签确认
- **小扣子**完成阶段性任务时，主动汇报进度与产物
- **小扣子**遇到卡点时，立即反馈给生哥，不阻塞等待

### 汇报模板

```markdown
## 任务：[任务名]
- 状态：✅ 已完成 / 🔄 进行中 / ⚠️ 卡点
- 产出：[文件路径 / 链接]
- 关键决策：[决策点与依据]
- 待确认：[需要生哥确认的事项]
- 下一步：[下一步计划]
```
