## 一、核心工作模型：Agent Loop（闭环验证循环）

```
生成 → 检查（客观标准） → 反馈（结果塞回上下文） → 重试
```

**关键区别**：
- ❌ 大多数AI工具只做第一步"生成"，然后交给人
- ✅ Codex做完整闭环：用测试/lint/业务规则客观验证，不靠AI自己判断"我觉得不错"
- 循环持续直到：退出条件满足（测试通过/达到最大重试/用户终止）

**风声落地**：
- 每条工单必须有验收标准（true/false可判定），不是"看着差不多"
- 品控不是终点检查，是**每一步都可验证**的闭环
- 小酒窝儿品控=自动触发，不靠人盯着

## 二、三级任务架构：Session→Task→Turn

```
Session（会话配置+状态）
  └── Task（一次用户输入的工作单元，同一时刻最多1个）
       └── Turn（一轮"请求→执行→产出"的迭代）
```

- **Task** = 工单（有明确输入、交付物、验收标准）
- **Turn** = 工单内的一次迭代（尝试→验证→修正）
- 一个Task可以跑多个Turn，直到验收标准=true

**风声落地**：
- 我们的工单=Task，催办/修正=Turn
- 工单模板强制选（已落地），验收标准必须是true/false
- 一次只推进一个核心Task，不并行散焦

## 三、安全边界硬编码，不靠AI自觉

Codex最被低估的设计：权限策略、网络策略、守卫机制**全部在Agent循环之外**。

即使提示词写得再好，边界也是硬编码的——不靠AI"自觉"遵守规则。

**风声落地**：
- 结构化工单模板=代码级强制（空白Issue禁提交，已落地）
- 品牌定位一字不改=硬规则，不靠记忆
- 对外内容铁律=必须代码级拦截，不能只靠提示词
- 未来方向：关键规则从MEMORY.md迁到代码/配置层强制执行

## 四、工具系统三层解耦

```
模型返回FunctionCall
  ↓ ToolRouter（协议解析层：把模型返回解析成ToolCall）
  ↓ ToolRegistry（路由分发层：按名称找到handler）
  ↓ Handler（业务执行层：真正执行）
```

**风声落地**：
- Agent分工=风声的ToolRouter：小鱼儿接收指令→路由到对应Agent
- 每个Agent=一个Handler：只负责自己的领域，不越界
- 新增能力=新增Handler，不改核心路由逻辑

## 五、Skills模块化：可复用能力单元

Codex Skills = SKILL.md + scripts/ + references/ 三件套

- 可搜索、可安装、可跨Agent复用
- 一个Skill封装完整的工作流，不依赖特定Agent

**风声落地**：
- 风声Skill标准化（三件套格式）
- 品控检查Skill、数据抓取Skill、知识萃取Skill——可复用，不绑死某个Agent
- ClawHub/红狐数据的Skill生态可直接借鉴

## 六、从聊天到干活：关键范式转变

| 聊天模式 | 干活模式 |
|---------|---------|
| 生成内容交给人 | 生成→验证→修正→交付 |
| 靠AI自觉遵守规则 | 硬编码强制执行 |
| 一次对话=一次输出 | 一次Task=多轮Turn闭环 |
| 工具是辅助 | 工具是执行主体 |
| 人盯着每一步 | 人设验收标准，Agent自己跑到通过 |

## 七、风声数字团队对标行动

| Codex做法 | 风声现状 | 行动项 |
|----------|---------|-------|
| 闭环验证 | 品控人工触发 | 品控对接CI/CD，PR自动触发检查 |
| 权限硬编码 | 铁律靠MEMORY.md | 关键规则迁到代码层强制 |
| Skills标准化 | Skill零散 | 推三件套标准（SKILL.md+scripts+references） |
| 多Agent并行 | 经常串行等 | 工单并行执行，不排队 |
| 客观验收 | 有时"看着差不多" | 验收标准必须true/false |

## 八、记忆口诀

**生成要检查，检查靠客观，客观才闭环，闭环才交付。**

规则不靠自觉，靠硬编码。
工单不是聊天，是Task+Turn。
验收不是感觉，是true/false。
