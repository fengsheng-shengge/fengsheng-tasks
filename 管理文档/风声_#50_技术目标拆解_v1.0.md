## 一、文档背景

### 1.1 从 #49 验证结论出发

Issue #49 目标倒推体系验证通过了 4 个维度：

| 维度 | 结论 | 对技术的影响 |
|------|------|------------|

| 产品可行性 | 5 产品各有独立闭环路径 | 每个产品需要独立的埋点/反馈/留存动作 |

## 三、核心技术架构设计

### 3.1 数据流转总览

```
用户访问 → tracker.js (浏览器) → /api/event (Cloudflare Worker)
                                               ↓
                                           D1 数据库
                                               ↓
                                    Dashboard 聚合查询
                                               ↓
                                    运营侧可视化分析
```

### 3.2 关键技术选型

| 领域 | 方案 | 原因 |
|------|------|------|
| 前端 | 原生 HTML/CSS/JS | 轻量、首屏快、无构建依赖 |
| 后端 | Cloudflare Workers + D1 | 零运维、边缘部署、免费额度够用 |
| 缓存 | Cloudflare KV | 与 Worker 生态无缝集成 |
| 埋点 | 自研 tracker.js | 数据主权在手，不依赖第三方 |
| 部署 | Cloudflare Pages | 静态资源 CDN 化 |
| 版本管理 | Git + GitHub Actions | CI 自动化，PR 必经 CI 检查 |

## 五、风险与缓解措施

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|
| Cloudflare token 再次失效 | 中 | 高 | token 存入 GitHub Secrets + 健康检查告警 |
| 业务侧输入延迟 | 高 | 中 | 技术侧先做占位版本，业务输入到位后替换 |
| 真实用户数据太少 | 中 | 低 | Dashboard 空状态友好提示 + 前 2 周主动邀请种子用户 |
| SEO 收录慢 | 中 | 中 | sitemap 提前提交 + 小红书/抖音作为冷启动主渠道 |
